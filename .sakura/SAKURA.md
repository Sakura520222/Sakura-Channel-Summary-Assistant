# Sakura-Bot 项目概述

## 项目简介
Sakura-Bot 是一款基于 AI 的智能 Telegram 频道管理工具，面向频道管理员提供自动内容总结、智能问答、投票生成与历史分析能力，帮助提升社群运营效率。

## 技术栈
- **语言**：Python 3.13+
- **交互**：Telegram Bot API
- **AI**：OpenAI 兼容 API（DeepSeek、OpenAI 等）+ RAG 检索增强生成 + 独立提示词系统
- **调度**：内置定时器（每日、每周等）
- **存储**：文件/轻量存储，对话总结持久化
- **测试**：pytest + 集成测试 + 覆盖率分析
- **质量**：Ruff、Black、pre-commit
- **部署**：Docker & Docker Compose，跨平台启动脚本
- **配置**：dotenv 环境变量 + JSON 配置文件，统一由 `core/config.py` 加载

## 架构设计与关键决策
项目按功能划分 `core/` 子包，AI、命令、配置、事件处理、转发等职责分离。`main.py` 与 `qa_bot.py` 作为不同服务入口按需加载模块。

**关键决策与已知架构债务**：
- **分层模糊**：`core.telegram` 直接包含业务工具函数，`core.services` 通过延迟导入反向依赖。理想方向是将通用逻辑下沉至 `core.services.poll` 或无依赖的 `core.utils`。
- **配置双通道**：环境变量 + JSON 热更新并存，但类型转换逻辑分散，缺乏统一的布尔值解析工具，已导致多次不一致。
- **延迟导入**：用于打破循环依赖，但增加了依赖关系复杂度。Telethon 特定请求的延迟导入仍散布在多个函数体内，未集中管理。
- **死代码残留风险**：移除功能模块（如 SQLite 适配器）时需全局搜索旧适配器名称和依赖文件，避免遗留未调用方法或导入残留。
- **启动检查逻辑分散**：多处模块独立实现“数据库是否就绪”检查，建议提取为统一健康状态服务。
- **版本号管理债务**：多处硬编码版本号已形成模式，必须强制单点定义（仅 `core/__init__.py`）。
- **仓储模式反模式**：全局单例仓储 + 无依赖传递导致单元测试困难，应强制依赖注入重构。
- **消息格式化与发送耦合**：`_format_*` 方法与 `parse_mode` 决策分散在不同位置，缺少统一的“安全 Markdown 消息构建器”。

## 已知问题和注意事项
1. **配置布尔陷阱**：`bool('false')` 在 Python 中为 `True`，统一解析函数 `parse_bool` 需显式处理 `None`、`int`、`str` 类型。
2. **写入/读取不对称**：动态配置写入可能持久化字符串类型的布尔值，导致“治标不治本”。
3. **异常屏蔽**：`finally` 块中裸调用清理函数，若清理过程抛出异常会覆盖原始业务异常。
4. **构建后覆盖反模式**：先构造对象再强制覆盖属性，削弱内聚性。
5. **配置变更全链路风险**：新增配置项时，需全程追踪类型转换与默认值一致性。
6. **时区比较不一致**：`datetime` 对象 naive 与 aware 混用会抛 `TypeError`，需显式处理。
7. **空值保护遗漏**：调用 LLM 或外部 API 时，空值检查未覆盖所有出现位置。
8. **异常静默处理反模式**：多处 `except` 块仅返回空值而不记录日志，形成“逻辑地雷”。
9. **延迟导入零散**：函数内延迟导入存在于循环或高频路径，缺乏注释说明原因。
10. **提示词缓存过期风险**：注入的 `current_time` 可能过期，导致 AI 使用过时上下文。
11. **文档与实现偏离**：抽取方法时 docstring 与实际行为不一致。
12. **字符串拼接空值过滤缺失**：使用 `" ".join(list)` 构建消息时，需主动过滤 falsy 值，避免多余空格。
13. **全局配置变量直接引用风险**：模块级配置变量用于 URL 拼接时，缺少非空守卫和日志记录。
14. **深链接用户名空值**：构造 `https://t.me/` 链接时，需检查目标用户名是否为空，并记录警告。
15. **多行字符串拼接一致性**：`\n` 与 `join()` 混用时，应对每个元素调用 `strip()` 或调整列表顺序避免前后空格。
16. **异步方法 docstring 缺失规范**：异步方法未强制要求在 docstring 首行标注“(异步)”或“Async”。
17. **白名单/黑名单扩展遗忘风险**：显式集合匹配或前缀匹配时，缺少添加注释说明“当前类型列表及未来新增类型的处理预期”。
18. **parse_mode 与内容安全耦合**：启用 Markdown 时动态内容必须转义，否则特殊字符会导致解析异常。
19. **数据库适配器移除不完整**：移除 SQLite 时未同步清理 `requirements.txt` 依赖、全局导入残留、迁移脚本版本链完整性。
20. **版本号硬编码副本**：`core/__init__.py` 之外的其他文件直接写 `"1.x.x"` 字符串，违反 DRY 规范。
21. **循环体内常量创建**：在 `for` 或 `while` 循环体中创建内容不变的 `frozenset`、`dict`，应提升到模块级别或循环外部。
22. **业务类型名字符串散落**：如 `"submission_approved"` 在多处使用硬编码字符串，应改用 `Enum` 或模块级常量统一管理。
23. **简化逻辑的异常行为不一致**：将 if-else 简化为单行表达式时，未对比简化前后的异常类型和传播路径，可能改变调用方预期。

## 审查中发现的重要模式
- **配置解析防御模式**：对字符串布尔值先做语义解析，再回退 `bool()`，是有效的松散配置值解析方法。
- **异步资源泄漏防范**：`nonlocal` 标志位 + `finally` 清理模式可有效防止事件处理器泄漏。
- **增量审查的上下文锚定与横向扫描**：依托历史审查记录定位问题，主动扫描相似模式捕获 diff 盲区中的重复逻辑。
- **防御性编程渐进增强**：从依赖异常兜底转向主动校验和语义正确处理，提升代码健壮性。
- **注释作为弱断言**：通过前置守卫保证非 `None` 后，用注释说明保证逻辑，是务实的折中方案。
- **全量重审优势识别**：对于超过 2000 行或新增核心模块的大型变更，全量重审能发现重复模式、跨文件集成问题和测试盲区。
- **显式白名单 + 默认降级**：安全优先，未明确声明的类型默认使用纯文本，防止 Markdown 解析崩溃。
- **注释驱动的策略说明**：核心逻辑策略通过注释而非代码结构体现，可读性好但维护性脆弱。
- **“集合白名单”模式**：使用 `frozenset` 显式枚举合法值，比隐式前缀匹配更确定，应统一用于支持的 `parse_mode` 类型等场景。
- **常量提升缺失反模式**：函数循环体内定义不可变常量，反映对 Python 对象生命周期关注不足。

## 团队约定和规范
### 代码与测试
- 强制使用 Black 格式化、Ruff 静态检查，pre-commit 自动执行。
- pytest 测试遵循 `test_*.py` 命名，共享固件在 `conftest.py`，要求运行覆盖率分析。
- 模块化设计，`core/` 内按功能分包，禁止跨核心层级反向依赖。

### 配置管理
- 敏感凭证通过 `.env` 注入，非敏感项使用 JSON；所有配置经 `core/config.py` 校验与加载。
- **布尔解析强制规范**：任何从外部源读取的布尔配置，必须使用项目级统一函数 `parse_bool(value, default)`，禁止直接 `bool()` 或手写 `isinstance` 逻辑。该函数必须显式处理 `None`、`int`、`str` 类型，并对未知字符串值记录警告。
- **写入对称性规则**：支持动态写入的配置项，写入侧必须持久化正确的预期类型，审查时需验证写入/读取逻辑对称。
- **新增配置项审查**：必须全链路追踪类型转换与默认值一致性。
- **全局配置变量引用规则**：直接使用模块级配置变量时，必须添加非空守卫并记录警告日志，禁止假设变量已配置。
- **配置热重载类型安全**：`update_module_variables` 等热更新函数中，必须对每个配置值调用类型转换函数，不能直接赋值。

### 数据库与适配器变更规范
- 移除数据库适配器时，必须检查 `requirements.txt` / `pyproject.toml` 依赖同步清理，全局搜索旧适配器名称（正则 `sqlite|aiosqlite`），验证迁移脚本版本链完整性，并确认 CHANGELOG + 升级指南更新。
- **简化逻辑的异常兼容性检查**：将 if-else 简化为单行表达式时，必须对比简化前后的异常类型和传播路径，确保行为一致，若改变需在 PR 中标注 breaking change。

### 异常与资源清理
- `finally` 块中的清理调用必须包裹 `try/except` 并记录日志，禁止裸调用。
- 异步资源注册与清理须使用标志位防护，确保任何退出路径均执行清理。
- **异常处理必须日志**：任何 `except` 块不得仅返回空值/默认值，必须至少记录 `logger.warning` 信息。

### 数据处理与防御
- **时区比较一致性**：`datetime` 对象比较时必须确保 naive/aware 一致，否则应显式附加时区或拒绝比较并记录警告。
- **空值保护全局一致性**：调用 LLM 或外部 API 后访问嵌套字段时，应在所有出现位置检查有效性。
- **延迟导入集中**：延迟导入统一放置在模块顶部或专门管理区域；若因循环依赖必须在函数内导入，应置于函数体最开始位置并加注释说明原因，严禁夹在业务逻辑中或出现在循环体内。
- **实体标识符统一提取**：获取 Telegram 频道/用户可读标识符必须使用统一工具函数，禁止各处手写回退模式；交叉文件出现≥2次即强制抽取。
- **字符串拼接过滤规范**：使用 `" ".join(list)` 构建用户可见内容时，必须主动过滤 `None`、空字符串和 `False` 值。`\n` 与 `join()` 混用时，应对每个元素调用 `strip()` 或调整列表顺序。
- **深链接空值检查**：涉及 `startswith("https://t.me/")` 等深链接构造的代码，必须检查目标用户名/ID 是否为空，并添加警告日志。
- **日期时间处理规则**：对未知来源的 `datetime` 或字符串，统一使用 `parse_date` 工具函数，封装类型判断、字符串解析和错误处理。
- **兜底逻辑显式化**：使用 `else: str(obj)[:N]` 作为最终兜底时，必须记录 `logger.warning` 或添加注释，说明可能产生非预期输出。
- **常量定义位置规则**：若一个变量是不依赖于实例或函数参数的不变常量，必须在模块级别定义。函数/方法内部禁止定义 `frozenset`、`tuple`、`dict` 等不可变常量（除非值依赖于运行时动态计算）。
- **字符串类型名转枚举规则**：当业务类型名出现在两个或以上不同位置时，必须使用 `Enum` 或模块级常量统一管理，集合匹配时必须引用枚举成员。

### AI 集成
- 提示词与人格设定独立存放于 `data/` 目录，便于版本管理和调优。
- AI 客户端抽象在 `core/ai`，支持多后端，新增模型只需扩展配置。
- 若提示词模板存在缓存，需确保注入的动态信息不会过期，或设置合理的失效机制。

### Telegram 消息格式规范
- 所有通过 Telegram Bot API 发送的消息，必须在发送函数前明确注释 `parse_mode` 的值及理由。
- 若 `parse_mode` 非固定值，必须提供 `_get_parse_mode` 工具函数并编写文档字符串，说明每种类型对应的模式。
- 禁止在未确认情况下混用 `MarkdownV2` 和 `HTML` 语法。
- 审查时必须交叉检查：消息内容中的特殊字符是否被转义、`parse_mode` 与格式符号是否匹配。
- **用户可见字符串四联检查**：代码逻辑 + i18n key 存在性 + i18n 内容 + 注释示例 → 四者一致。禁止硬编码中文/英文字符串而未通过 `get_text` 调用。

### 文档与流程
- 中英文 README，`wiki/` 沉淀架构与维护记录，重要变更记入 `CHANGELOG.md`。
- 贡献流程见 `CONTRIBUTING.md`，PR 需通过 CI 检查。
- **增量审查规范**：优先采用增量审查闭环策略，审查时主动扫描上下文相似模式并横向对比已知热点调用；比对 PR 描述与变更文件列表，检查关联资源文件是否同步修改；对修复类 PR 显式验证行为等价性。增加最小横向检查清单：变更函数的上层调用方、同模块下相同模式问题、上一轮未修复问题。对于涉及配置变量的变更，应强制执行“配置使用点全量扫描”。功能移除类 PR 必须扩大边界检查至 `tests/`、`docs/`、`.github/`。
- **大型变更全量重审**：对于超过 2000 行或新增核心模块的 PR，强制执行一次全量重审，发现重复模式、跨文件集成和测试盲区。
- **docstring 一致性**：抽取或新增函数时，必须核对 docstring 描述与实际返回类型、副作用一致。异步方法 docstring 必须在前 10 个字符内包含“异步”或“Async”。
- **三联检查清单**：对于底栏/消息模板类修改，必须检查三处一致性：代码逻辑、i18n 字符串、注释/文档。
- **历史遗留问题提醒**：增量审查中，若 PR 描述或变更文件未涉及上一轮标记的“未修复问题”，审查报告末尾需增加“遗留风险提醒”段落，包含具体 Issue 编号。
- **提交信息交叉验证**：当 PR 描述中声明“修复 Issue A/B/C”时，审查者必须逐项确认相关文件是否在变更范围内，若不在需明确标记为“声明未实现”并建议修改 PR 描述。
- **审查报告质量门禁**：审查报告中的“已修复”必须指向具体 commit hash，不能依赖 PR 描述。历史问题累计超过 3 次反思未修复则升级为架构债会议议题。

### 代码组织约束
- 同一转换/工具函数在同一文件内出现超过 2 次，必须抽取为私有辅助函数；跨文件同一模式≥2次，应考虑抽取至公共层。
- 跨层导入方向：`core.telegram` → `core.services` 为允许方向，反向依赖必须有文档说明并计划迁移到公共模块。
- 被多模块使用的构建函数应归属 `core.services.poll` 或 `core.utils`，不得留在 `core.telegram`。
- **DRY 版本号规范**：版本号必须在且仅在 `core/__init__.py` 中定义，其他文件通过 `from core import __version__` 引用，禁止硬编码副本。Lint 规则应禁止在其他文件出现 `"1.x.x"` 模式。
- **条件兜底日志**：当 `if-elif-else` 中 `else` 分支采用兜底行为，必须添加 `logger.debug` 或注释说明该兜底是设计选择而非遗漏。
- **全局状态访问显式依赖**：`get_xxx_repo()` 等函数必须显式接受依赖参数（如 `db_manager`），禁止隐式依赖全局单例，且强制要求在返回前验证实例可用性，否则抛 `RuntimeError`。
- **依赖同步检查**：移除任何第三方库或内置模块支持时，PR 必须同时更新 `requirements.txt` 或 `pyproject.toml`。
