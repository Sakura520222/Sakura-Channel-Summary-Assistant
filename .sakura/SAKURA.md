# Sakura-Bot 项目概述

## 项目简介
Sakura-Bot 是一款基于 AI 的智能 Telegram 频道管理工具，面向频道管理员提供自动内容总结、智能问答、投票生成与历史分析能力，帮助提升社群运营效率。

## 技术栈
- **语言**：Python 3.13+
- **交互**：Telegram Bot API
- **AI**：OpenAI 兼容 API（DeepSeek、OpenAI 等）+ RAG 检索增强生成 + 独立提示词系统
- **调度**：内置定时器（每日、每周等）
- **存储**：文件/轻量存储，对话总结持久化
- **测试**：pytest + 集成测试 + 覆盖率分析
- **质量**：Ruff、Black、pre-commit
- **部署**：Docker & Docker Compose，跨平台启动脚本
- **配置**：dotenv 环境变量 + JSON 配置文件，统一由 `core/config.py` 加载

## 架构设计与关键决策
项目按功能划分 `core/` 子包，AI、命令、配置、事件处理、转发等职责分离。`main.py` 与 `qa_bot.py` 作为不同服务入口按需加载模块。

**关键决策与已知架构债务**：
- **分层模糊**：`core.telegram` 直接包含业务工具函数，`core.services` 通过延迟导入反向依赖，形成跨层耦合。理想方向是将通用逻辑下沉至 `core.services.poll` 或无依赖的 `core.utils`。
- **配置双通道**：环境变量 + JSON 热更新并存，但类型转换逻辑分散，缺乏统一的布尔值解析工具，已导致多次不一致。
- **延迟导入**：用于打破循环依赖，但增加了依赖关系复杂度和重构难度。Telethon 特定请求（如 `GetFullChannelRequest`）的延迟导入仍散布在多个函数体内，未集中管理。
- **死代码清理**：已移除 `agent_loop.py` 并将循环逻辑内聚到已有模块，减少冗余入口，体现对架构简洁性的追求。
- **工具函数碎片化**：获取 Telegram 实体标识符的模式（`get_entity → hasattr(username) → str(id)`）在多处重复，缺少统一的 `get_channel_display_identifier(client, entity_id)` 工具函数，是需收敛的技术债务。

## 已知问题和注意事项
1. **配置布尔陷阱**：`bool('false')` 在 Python 中为 `True`，从 JSON 热更新或环境变量读取的字符串 `"false"` 会得出错误结果。代码中多处手写 `isinstance` 转换，逻辑不一致，极易复发。
2. **写入/读取不对称**：动态配置写入可能持久化字符串类型的布尔值，即使读取侧已修正，配置文件仍残留错误类型，导致“治标不治本”。
3. **异常屏蔽**：`finally` 块中裸调用清理函数，若清理过程抛出异常会覆盖原始业务异常，严重妨碍排查。
4. **构建后覆盖反模式**：先构造对象再强制覆盖属性，削弱内聚性，应优先参数化构建。
5. **配置变更全链路风险**：新增配置项时，若未从环境变量→模块变量→热更新→持久化加载→业务使用全程追踪类型与默认值一致性，容易引入隐蔽缺陷。
6. **时区比较不一致**：`datetime` 对象比较若 naive 与 aware 混用会抛 `TypeError`，当前部分代码依赖外层 except 静默返回空结果，掩盖错误并丢失诊断信息。
7. **空值保护遗漏**：调用 LLM 或外部 API 访问 `response.choices` 等路径时，空值检查未覆盖所有出现位置，存在不一致风险。
8. **异常静默处理反模式**：多处 `except` 块仅返回空值或默认值而不记录任何日志，形成“逻辑地雷”，后续排查困难。
9. **延迟导入零散**：函数内延迟导入存在于循环或高频路径，未遵循集中管理惯例；部分延迟导入夹在业务逻辑中间，缺乏注释说明原因，影响可维护性。
10. **提示词缓存过期风险**：若 QA 引擎的提示词模板被缓存，注入的 `current_time` 可能过期，导致 AI 使用过时上下文，需关注缓存失效策略。
11. **文档与实现偏离**：抽取方法时 docstring 描述可能与实际行为（如字符串处理细节、返回值中 `None` 的可能性）不一致，降低代码可信度。

## 审查中发现的重要模式
- **配置解析防御模式**：对字符串布尔值先做语义解析（`lower()` 与 `'true'`/`'false'` 比较），再回退 `bool()`，是有效的松散配置值解析方法。
- **异步资源泄漏防范**：`nonlocal` 标志位 + `finally` 清理模式可有效防止事件处理器泄漏，需确保标志位在注册前设置，且任何提前 `return` 均不绕过清理。
- **增量审查的上下文锚定与横向扫描**：依托历史审查记录快速定位问题，并通过主动扫描相似模式（尤其 `get_entity` 等热点调用）捕获 diff 盲区中的重复逻辑。此“diff + 上下文扫描”方式应作为最佳实践保留。
- **防御性编程渐进增强**：从依赖异常兜底转向主动校验和语义正确处理，如增加空值保护、显式处理时区、替换 `asyncio.get_running_loop()` 等，提升代码健壮性。
- **修复引入新变数需高亮**：日期过滤改为 `datetime` 对象比较时，自然引入时区混用新风险，审查中已主动追溯此类边缘情况，应作为常态策略。
- **审查驱动修复与 DRY 抽取意识**：开发者能快速、精确响应审查意见，一次性修复多个独立问题，并以私有辅助方法消除重复，表明团队对“≥2次即抽取”规范的认同度提升。
- **注释作为弱断言**：通过前置守卫保证非 None 后，用注释说明保证逻辑，替代显式断言，在守卫简单清晰时是务实的折中方案。
- **重构行为等价性验证**：审查提取方法时不仅检查结构，更强调行为与重构前完全一致，并对 docstring 做即时一致性核对。

## 团队约定和规范
### 代码与测试
- 强制使用 Black 格式化、Ruff 静态检查，pre-commit 自动执行。
- pytest 测试遵循 `test_*.py` 命名，共享固件在 `conftest.py`，要求运行覆盖率分析。
- 模块化设计，`core/` 内按功能分包，禁止跨核心层级反向依赖。

### 配置管理
- 敏感凭证通过 `.env` 注入，非敏感项使用 JSON；所有配置经 `core/config.py` 校验与加载。
- **布尔解析强制规范**：任何从外部源读取的布尔配置，必须使用项目级统一函数 `parse_bool(value, default)`，禁止直接 `bool()` 或手写 `isinstance` 逻辑。
- **写入对称性规则**：支持动态写入的配置项，写入侧必须持久化正确的预期类型，审查时需验证写入/读取逻辑对称。
- 新增配置项审查必须全链路追踪类型转换与默认值一致性。

### 异常与资源清理
- `finally` 块中的清理调用必须包裹 `try/except` 并记录日志，禁止裸调用。
- 异步资源注册与清理须使用标志位防护，确保任何退出路径均执行清理。
- **异常处理必须日志**：任何 `except` 块不得仅返回空值/默认值，必须至少记录 `logger.warning` 信息。

### 数据处理与防御
- **时区比较一致性**：`datetime` 对象比较时必须确保 naive/aware 一致，否则应显式附加时区或拒绝比较并记录警告，禁止依赖空异常兜底。
- **空值保护全局一致性**：调用 LLM 或外部 API 后访问 `response.choices`、`response.content` 等路径时，应在所有出现位置检查有效性。
- **延迟导入集中**：延迟导入统一放置在模块顶部或专门管理区域；若因循环依赖必须在函数内导入，应置于函数体最开始位置并加注释说明原因，严禁夹在业务逻辑中或出现在循环体内。
- **闭包与 `run_in_executor` 风格**：若传递给 `run_in_executor` 的逻辑超过一行或涉及外部变量，必须使用具名局部函数。
- **实体标识符统一提取**：获取 Telegram 频道/用户可读标识符（用户名或 ID）必须使用统一工具函数，禁止各处手写 `getattr` 回退模式；交叉文件出现≥2次即强制抽取。
- **不变性注释**：当局部变量通过前置 `return` 保证非 `None` 后使用，鼓励添加简短注释（如 `# guaranteed non-None by guard above`），提高可读性。

### AI 集成
- 提示词与人格设定独立存放于 `data/` 目录，便于版本管理和调优。
- AI 客户端抽象在 `core/ai`，支持多后端，新增模型只需扩展配置。
- 若提示词模板存在缓存，需确保注入的动态信息（如时间）不会过期，或设置合理的失效机制。

### 文档与流程
- 中英文 README，`wiki/` 沉淀架构与维护记录，重要变更记入 `CHANGELOG.md`。
- 贡献流程见 `CONTRIBUTING.md`，PR 需通过 CI 检查。
- **增量审查规范**：优先采用增量审查闭环策略，审查时主动扫描上下文相似模式并横向对比已知热点调用；比对 PR 描述/提交信息与实际变更文件列表，检查关联资源文件是否同步修改；对修复类 PR 显式验证行为等价性。
- **docstring 一致性**：抽取或新增函数时，必须核对 docstring 描述与实际返回类型、副作用一致，尤其关注大小写、空值可能性及异常。

### 代码组织约束
- 同一转换/工具函数在同一文件内出现超过 2 次，必须抽取为私有辅助函数；跨文件同一模式≥2次，应考虑抽取至 `core.utils` 或 `core.services.poll` 等公共层。
- 跨层导入方向：`core.telegram` → `core.services` 为允许方向，反向依赖必须有文档说明并计划迁移到公共模块。
- 被多模块使用的构建函数应归属 `core.services.poll` 或 `core.utils`，不得留在 `core.telegram`。
