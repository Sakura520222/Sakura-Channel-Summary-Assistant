# Sakura-Bot 项目概述

## 项目简介
Sakura-Bot 是一款基于 AI 的智能 Telegram 频道管理工具，面向频道管理员提供自动内容总结、智能问答、投票生成与历史分析能力，帮助提升社群运营效率。

## 技术栈
- **语言**：Python 3.13+
- **交互**：Telegram Bot API
- **AI**：OpenAI 兼容 API（DeepSeek、OpenAI 等）+ RAG 检索增强生成 + 独立提示词系统
- **调度**：内置定时器（每日、每周等）
- **存储**：文件/轻量存储，对话总结持久化
- **测试**：pytest + 集成测试 + 覆盖率分析
- **质量**：Ruff、Black、pre-commit
- **部署**：Docker & Docker Compose，跨平台启动脚本
- **配置**：dotenv 环境变量 + JSON 配置文件，统一由 `core/config.py` 加载

## 架构设计与关键决策
项目按功能划分 `core/` 子包，AI、命令、配置、事件处理、转发等职责分离。`main.py` 与 `qa_bot.py` 作为不同服务入口按需加载模块。

**关键决策与已知架构债务**：
- **命令元数据 + Handler 注册表模式**：声明式设计，集中描述命令参数、类型、确认需求，通过映射表关联处理函数，易于扩展但缺乏参数类型安全。超过 5 个工具的类应使用注册表字典映射。
- **配置双通道**：环境变量 + JSON 热更新并存，但类型转换逻辑分散，缺乏统一的布尔值解析工具，已导致多次不一致。
- **延迟导入**：用于打破循环依赖，但已演变为滥用，应集中到统一获取函数或使用 `lazy_import` 工具。
- **文件作为持久化存储**：直接读写 JSON 文件，无数据库或锁机制，成为并发竞态风险点。模块级 `asyncio.Lock` 仅保护单进程内并发，多 worker 下必须使用 `filelock` 或 Redis 分布式锁。
- **审计系统设计**：非阻塞写入 + 失败仅记录日志，符合可观测性最佳实践。
- **仓储模式反模式**：全局单例仓储 + 无依赖传递导致单元测试困难，应强制依赖注入重构。
- **版本号管理债务**：多处硬编码版本号，必须强制单点定义（仅 `core/__init__.py`）并加入 CI 检查。
- **参数清洗一致性**：多处出现 `params.get("key") or None` 将空字符串转 `None`，应统一为工具函数 `sanitize_optional_str`。
- **两段式验证结构**：先验证“两者都有”，再验证“只有一个”，最后隐式走“两者都无”分支，是清晰的替代嵌套 `if not A or not B` 的模式。
- **双存储对称性要求**：若设计两个平行的存储结构（如 `_result_store` 和 `_doc_result_store`），必须提供对称的增删改查接口。

## 已知问题和注意事项
1. **配置布尔陷阱**：`bool('false')` 为 `True`，统一使用 `parse_bool` 并显式处理 `None`、`int`、`str`。
2. **文件 I/O 安全漏洞**：禁止全量读取大文件；API 响应必须脱敏路径，仅返回文件名或相对路径。
3. **写入/读取不对称**：动态配置写入可能持久化字符串布尔值。
4. **异常屏蔽**：`finally` 块中避免调用可能抛异常的清理函数。
5. **构建后覆盖反模式**：先构造对象再强制覆盖属性，削弱内聚性。
6. **配置变更全链路风险**：新增配置项需全程追踪类型转换与默认值一致性。
7. **时区比较不一致**：naive 与 aware 类型混用会抛 `TypeError`。
8. **空值保护遗漏**：调用 LLM 或外部 API 后需检查所有嵌套字段。
9. **异常静默处理**：禁止 `except` 块仅返回空值不记录日志。
10. **延迟导入零散**：函数内延迟导入需加注释，避免在高频路径使用。
11. **提示词缓存过期风险**：注入的 `current_time` 可能过期。
12. **文档与实现偏离**：抽取方法时需同步更新 docstring。
13. **字符串拼接空值过滤**：使用 `" ".join(list)` 前需主动过滤 falsy 值。
14. **全局配置变量直接引用**：需添加非空守卫和日志。
15. **深链接用户名空值**：构造 `https://t.me/` 链接需检查目标是否为空。
16. **多行字符串拼接一致性**：`\n` 与 `join()` 混用时需对每个元素 `strip()`。
17. **异步方法 docstring 缺失规范**：异步方法需在 docstring 首行标注“(异步)”。
18. **白名单/黑名单扩展遗忘**：显式集合匹配需添加“预期处理方式”注释。
19. **parse_mode 与内容安全耦合**：启用 Markdown 时动态内容必须转义，推荐 `parse_mode="MarkdownV2"`。
20. **空字符串 vs None 语义**：Web API 中需统一约定空字符串的语义。
21. **版本号硬编码副本**：`core/__init__.py` 之外禁止直接写版本号。
22. **循环体内常量创建**：不可变常量（`frozenset`、`dict`）必须提升到模块级。
23. **业务类型名字符串散落**：两个以上位置使用的字符串类型名必须改用 `Enum`。
24. **简化逻辑的异常行为不一致**：简化 if-else 时需对比异常类型。
25. **过滤后行数不足无提示**：日志读取“读取最近 N 行 + 按级别过滤”可能返回少于请求数量且无警告。
26. **删除接口语义不一致**：WebUI 与 Telegram 接口语义需统一或在 PR 中说明原因。
27. **配置热重载非原子**：`load_config()` + `update_module_variables()` 两步之间无锁，不支持并发。
28. **数据库查询性能隐患**：模糊查询（`LIKE %xxx%`）在大表场景下需确认索引，新增查询方法必须评估数据量上限并添加 `LIMIT`。
29. **并发安全盲区**：模块级 `dict` 作为协程间共享状态时必须使用 `asyncio.Lock` 保护。
30. **依赖版本范围关联**：针对特定框架警告的抑制逻辑，需在注释中注明 PTB 版本约束和升级风险。
31. **PTB 会话管理警告**：`ConversationHandler` 与 `CallbackQueryHandler` 配合 `per_message=False` 时，必须使用 `warnings.catch_warnings` 精确抑制 `PTBUserWarning`。

## 审查中发现的重要模式
- **声明式命令注册表**：审查重点在元数据与 handler 参数类型匹配，options 是否与内部校验重复。
- **防御性日志读取**：使用 `deque(maxlen=N)` 替代完整读取，是处理大文件的好模式。
- **异步资源泄漏防范**：`nonlocal` 标志位 + `finally` 清理模式可有效防止事件处理器泄漏。
- **增量审查的横向扫描**：必须检查变更函数的上层调用方、同模块相同模式问题、上一轮未修复问题（安全相关未修复须升级为严重问题）。
- **注释作为弱断言**：通过前置守卫保证非 `None` 后，用注释说明保证逻辑。
- **显式白名单 + 默认降级**：未明确声明的类型默认使用纯文本，防止 Markdown 解析崩溃。
- **常量提升缺失反模式**：函数循环体内定义不可变常量，反映对对象生命周期关注不足。
- **isawaitable 语义化**：用 `inspect.isawaitable` 替代 `hasattr` 检查是更规范的异步判断模式。
- **敏感信息脱敏演进**：精确匹配 + 后缀匹配（`_token`、`_secret`）比前缀匹配更安全。
- **“优先新语义 + 兼容旧语义”API 演进**：未使用 feature flag 时，必须用注释说明旧语义触发条件。

## 团队约定和规范
### 代码与测试
- 强制使用 Black 格式化、Ruff 静态检查，pre-commit 自动执行。
- pytest 测试遵循 `test_*.py` 命名，共享固件在 `conftest.py`，要求覆盖率分析。新增端点必须包含：正常路径、异常路径、边界条件。
- 模块化设计，`core/` 内按功能分包，禁止跨核心层级反向依赖。
- 异步代码必须使用 `pytest-asyncio` 编写并发安全测试。

### 配置管理
- 敏感凭证通过 `.env` 注入，非敏感项使用 JSON；所有配置经 `core/config.py` 校验。
- **布尔解析强制规范**：使用项目级统一函数 `parse_bool`，禁止直接 `bool()`。
- **新增配置项审查**：必须全链路追踪类型转换与默认值一致性。
- **全局配置变量引用规则**：必须添加非空守卫并记录警告。

### 文件 I/O 与安全规范
- **禁止全量读取未知大小文件**：使用流式或尾部读取。
- **API 响应禁止暴露服务器文件路径**：仅返回文件名或相对路径。
- **文件句柄泄露检查**：所有 `open()` 调用必须使用上下文管理器。
- **并发文件操作**：使用 `try/except` 捕获 `FileNotFoundError`，评估是否需要文件锁。

### 数据处理与防御
- **时区比较一致性**：确保 naive/aware 一致。
- **空值保护**：调用外部 API 后在所有出现位置检查有效性。
- **延迟导入集中**：若必须在函数内导入，置于函数体最开始并加注释。
- **字符串拼接过滤**：使用 `" ".join(list)` 前主动过滤 falsy 值。
- **常量定义位置**：函数内部禁止定义不可变常量。
- **字符串类型名转枚举**：出现在两个及以上位置时必须使用 `Enum`。
- **整数转换异常捕获**：从请求中提取整数参数时，必须使用 `try/except ValueError` 返回 400。

### 审查流程规范
- **增量审查强制检查清单**：搜索本次修改函数的所有调用方；检查上轮警告是否修复；对提取的辅助函数搜索是否可复用。
- **PR 描述-diff 一致性检查**：审查者必须逐条核对 PR 描述中的变更点是否在 diff 中体现。
- **版本号变更强制检查**：修改版本号是否仅在 `core/__init__.py` 中进行，CI 中应增加硬检查。
- **删除接口语义一致性**：Web API 与 Telegram 命令语义应一致，不一致须在 PR 中说明。
- **参数清洗一致性**：建议提取为 `get_optional_str(params, key)`。
- **警告抑制一致性检查**：新增 `warnings.catch_warnings` 时，必须 grep 项目其他相同警告类型并同步处理。
