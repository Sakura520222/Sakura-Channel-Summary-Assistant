# Sakura-Bot 项目概述

## 项目简介
Sakura-Bot 是一款基于 AI 的智能 Telegram 频道管理工具，面向频道管理员提供自动内容总结、智能问答、投票生成与历史分析能力，帮助提升社群运营效率。

## 技术栈
- **语言**：Python 3.13+
- **交互**：Telegram Bot API
- **AI**：OpenAI 兼容 API（DeepSeek、OpenAI 等）+ RAG 检索增强生成 + 独立提示词系统
- **调度**：内置定时器（每日、每周等）
- **存储**：文件/轻量存储，对话总结持久化
- **测试**：pytest + 集成测试 + 覆盖率分析
- **质量**：Ruff、Black、pre-commit
- **部署**：Docker & Docker Compose，跨平台启动脚本
- **配置**：dotenv 环境变量 + JSON 配置文件，统一由 `core/config.py` 加载

## 架构设计与关键决策
项目按功能划分 `core/` 子包，AI、命令、配置、事件处理、转发等职责分离。`main.py` 与 `qa_bot.py` 作为不同服务入口按需加载模块。

**关键决策与已知架构债务**：
- **命令元数据 + Handler 注册表模式**：声明式设计，集中描述命令参数、类型、确认需求，通过映射表关联处理函数，易于扩展但缺乏参数类型安全。
- **配置双通道**：环境变量 + JSON 热更新并存，但类型转换逻辑分散，缺乏统一的布尔值解析工具，已导致多次不一致。
- **延迟导入**：用于打破循环依赖，但已演变为滥用（单文件≥7处函数内导入），应集中到统一获取函数。
- **文件作为持久化存储**：直接读写 JSON 文件，无数据库或锁机制，成为并发竞态风险点（TOCTOU）。
- **审计系统设计**：非阻塞写入 + 失败仅记录日志，符合可观测性最佳实践，但失败路径未记录审计。
- **仓储模式反模式**：全局单例仓储 + 无依赖传递导致单元测试困难，应强制依赖注入重构。
- **版本号管理债务**：多处硬编码版本号（已累计6+处），必须强制单点定义（仅 `core/__init__.py`）并加入 CI 检查。

## 已知问题和注意事项
1. **配置布尔陷阱**：`bool('false')` 为 `True`，统一解析函数 `parse_bool` 需显式处理 `None`、`int`、`str`。
2. **文件 I/O 安全漏洞**：`Path.read_text()` 全量读取大日志文件造成 OOM；API 响应暴露服务器文件路径。
3. **写入/读取不对称**：动态配置写入可能持久化字符串布尔值。
4. **异常屏蔽**：`finally` 块中裸调用清理函数，若清理过程抛异常会覆盖原始业务异常。
5. **构建后覆盖反模式**：先构造对象再强制覆盖属性，削弱内聚性。
6. **配置变更全链路风险**：新增配置项需全程追踪类型转换与默认值一致性。
7. **时区比较不一致**：naive 与 aware 混用会抛 `TypeError`。
8. **空值保护遗漏**：调用 LLM 或外部 API 时空值检查未覆盖所有位置。
9. **异常静默处理反模式**：多处 `except` 块仅返回空值不记录日志，形成逻辑地雷。
10. **延迟导入零散**：函数内延迟导入存在于循环或高频路径，缺乏注释。
11. **提示词缓存过期风险**：注入的 `current_time` 可能过期。
12. **文档与实现偏离**：抽取方法时 docstring 与实际行为不一致。
13. **字符串拼接空值过滤缺失**：使用 `" ".join(list)` 需主动过滤 falsy 值。
14. **全局配置变量直接引用**：模块级配置变量用于 URL 拼接时，缺少非空守卫和日志。
15. **深链接用户名空值**：构造 `https://t.me/` 链接需检查目标用户名是否为空。
16. **多行字符串拼接一致性**：`\n` 与 `join()` 混用需对每个元素 `strip()`。
17. **异步方法 docstring 缺失规范**：异步方法未强制要求在 docstring 首行标注“(异步)”。
18. **白名单/黑名单扩展遗忘**：显式集合匹配缺少“当前类型列表及未来新增类型处理预期”注释。
19. **parse_mode 与内容安全耦合**：启用 Markdown 时动态内容必须转义。
20. **数据库适配器移除不完整**：移除 SQLite 时未同步清理依赖、导入残留、迁移脚本版本链。
21. **版本号硬编码副本**：`core/__init__.py` 之外直接写 `"1.x.x"`，违反 DRY。
22. **循环体内常量创建**：在循环中创建内容不变的 `frozenset`、`dict`，应提升到模块级。
23. **业务类型名字符串散落**：如 `"submission_approved"` 多处硬编码，应改用 `Enum`。
24. **简化逻辑的异常行为不一致**：将 if-else 简化为单行表达式时，未对比简化前后的异常类型。
25. **过滤后行数不足无提示**：日志读取“读取最近 N 行 + 按级别过滤”可能返回少于请求数量且无警告。
26. **删除接口语义不一致**：WebUI 按序号删除、Telegram 按 ID 删除，应在 PR 中说明原因。
27. **配置热重载非原子**：`load_config()` + `update_module_variables()` 两步之间无锁，不支持并发。

## 审查中发现的重要模式
- **声明式命令注册表**：审查重点放在元数据与实际 handler 参数类型是否匹配，options/choices 是否与 handler 内部校验逻辑重复。
- **防御性日志读取**：使用 `deque(maxlen=N)` 替代完整读取，是处理大文件的好模式，应推广。
- **异步资源泄漏防范**：`nonlocal` 标志位 + `finally` 清理模式可有效防止事件处理器泄漏。`aiofiles.open` 必须使用 `async with`。
- **增量审查的上下文锚定与横向扫描**：依托历史审查记录定位问题，主动扫描相似模式捕获 diff 盲区中的重复逻辑。必须检查：变更函数的上层调用方、同模块下相同模式问题、上一轮未修复问题。
- **防御性编程渐进增强**：从依赖异常兜底转向主动校验和语义正确处理。
- **注释作为弱断言**：通过前置守卫保证非 `None` 后，用注释说明保证逻辑。
- **全量重审优势**：对于超过2000行或新增核心模块的 PR，全量重审能发现重复模式、跨文件集成（如多组件重复初始化 localStorage）和测试盲区（如空 body 分支）。
- **显式白名单 + 默认降级**：未明确声明的类型默认使用纯文本，防止 Markdown 解析崩溃。
- **“集合白名单”模式**：使用 `frozenset` 显式枚举合法值，比隐式前缀匹配更确定。
- **常量提升缺失反模式**：函数循环体内定义不可变常量，反映对 Python 对象生命周期关注不足。
- **isawaitable 语义化**：用 `inspect.isawaitable` 替代 `hasattr` 检查是更规范的异步判断模式。

## 团队约定和规范
### 代码与测试
- 强制使用 Black 格式化、Ruff 静态检查，pre-commit 自动执行。
- pytest 测试遵循 `test_*.py` 命名，共享固件在 `conftest.py`，要求运行覆盖率分析。新增 API 端点必须包含：正常路径、异常路径（参数非法、依赖失败）、边界条件（空 body、超大输入）。
- 模块化设计，`core/` 内按功能分包，禁止跨核心层级反向依赖。

### 配置管理
- 敏感凭证通过 `.env` 注入，非敏感项使用 JSON；所有配置经 `core/config.py` 校验与加载。
- **布尔解析强制规范**：使用项目级统一函数 `parse_bool(value, default)`，禁止直接 `bool()`。必须显式处理 `None`、`int`、`str`，并对未知字符串值记录警告。
- **写入对称性规则**：动态写入配置项，写入侧必须持久化正确的预期类型。
- **新增配置项审查**：必须全链路追踪类型转换与默认值一致性。
- **全局配置变量引用规则**：直接使用模块级配置变量时，必须添加非空守卫并记录警告。
- **配置热重载类型安全**：`update_module_variables` 等热更新函数中，必须对每个配置值调用类型转换函数。

### 数据库与适配器变更规范
- 移除数据库适配器时，必须检查 `requirements.txt` / `pyproject.toml` 依赖、全局搜索旧适配器名称（正则）、验证迁移脚本版本链完整性、更新 CHANGELOG + 升级指南。
- **简化逻辑的异常兼容性**：将 if-else 简化为单行表达式时，必须对比简化前后的异常类型和传播路径，若改变需在 PR 中标注 breaking change。

### 文件 I/O 与安全规范
- **禁止全量读取未知大小文件**：如 `Path.read_text()`、`file.read()` 必须限制读取行数/字节数，使用流式或尾部读取。文件读取使用 `deque(maxlen=N)` + 逐行迭代。
- **API 响应禁止暴露服务器文件路径**：日志、配置等文件路径须脱敏（仅返回文件名或相对路径，如 `Path.name`）。
- **文件句柄泄露检查**：所有 `open()` 调用（包括 `aiofiles.open`）必须使用上下文管理器（`async with`），或在 `finally` 中显式关闭。
- **并发文件操作**：对文件系统进行“检查-读-改-写”时，禁止使用 `exists()` 预检查，必须使用 `try/except` 捕获 `FileNotFoundError`，并评估是否需要文件锁。

### 审计与异常规范
- **所有写操作必须记录审计**：且 try/except 包裹，失败时 `success=False` 也必须写入审计。审计记录包含操作前后的关键数据指纹（如删除行数、清理天数）。
- **异常处理必须日志**：任何 `except` 块不得仅返回空值/默认值，必须至少记录 `logger.warning`。
- `finally` 块中的清理调用必须包裹 `try/except` 并记录日志，禁止裸调用。
- 异步资源注册与清理须使用标志位防护，确保任何退出路径均执行清理。

### 数据处理与防御
- **时区比较一致性**：`datetime` 对象比较时必须确保 naive/aware 一致，否则显式附加时区或拒绝比较并记录警告。
- **空值保护全局一致性**：调用 LLM 或外部 API 后访问嵌套字段时，在所有出现位置检查有效性。
- **延迟导入集中**：统一放置在模块顶部或专门管理区域；若因循环依赖必须在函数内导入，置于函数体最开始并加注释，严禁出现在循环体内或夹在业务逻辑中。
- **实体标识符统一提取**：获取 Telegram 频道/用户可读标识符必须使用统一工具函数，禁止各处手写回退模式。
- **字符串拼接过滤规范**：使用 `" ".join(list)` 构建内容时，必须主动过滤 `None`、空字符串和 `False`。`\n` 与 `join()` 混用时，对每个元素 `strip()`。
- **深链接空值检查**：构造 `https://t.me/` 链接必须检查目标用户名/ID 是否为空，并添加警告日志。
- **日期时间处理**：对未知来源的 `datetime` 或字符串，统一使用 `parse_date` 工具函数。
- **兜底逻辑显式化**：使用 `else: str(obj)[:N]` 时，必须记录 `logger.warning` 或加注释。
- **常量定义位置**：不依赖实例或函数参数的不变常量，必须在模块级别定义。函数内部禁止定义 `frozenset`、`tuple`、`dict` 等不可变常量。
- **字符串类型名转枚举**：当业务类型名出现在两个及以上位置时，必须使用 `Enum` 或模块级常量统一管理。
- **整数转换异常捕获**：从请求中提取整数参数时，必须使用 `try/except ValueError` 捕获类型转换异常，返回 400 而非 500。

### AI 集成
- 提示词与人格设定独立存放于 `data/` 目录，便于版本管理和调优。
- AI 客户端抽象在 `core/ai`，支持多后端，新增模型只需扩展配置。
- 若提示词模板存在缓存，需确保注入的动态信息不会过期，或设置合理的失效机制。
- **经验系数可溯源**：任何魔数（如 `lines * 3`）应有注释说明推导过程和压测数据。

### Telegram 消息格式规范
- 所有通过 Telegram Bot API 发送的消息，必须在发送函数前明确注释 `parse_mode` 的值及理由。
- 若 `parse_mode` 非固定值，必须提供 `_get_parse_mode` 工具函数并编写文档字符串，说明每种类型对应的模式。
- 禁止在未确认情况下混用 `MarkdownV2` 和 `HTML` 语法。
- 审查时必须交叉检查：消息内容中的特殊字符是否被转义、`parse_mode` 与格式符号是否匹配。
- **用户可见字符串四联检查**：代码逻辑 + i18n key 存在性 + i18n 内容 + 注释示例 → 四者一致。禁止硬编码中文/英文字符串而未通过 `get_text` 调用。

### 代码组织约束
- 同一转换/工具函数在同一文件内出现超过 2 次，必须抽取为私有辅助函数；跨文件同一模式≥2次，考虑抽取至公共层。
- 跨层导入方向：`core.telegram` → `core.services` 为允许方向，反向依赖必须有文档说明并计划迁移。
- 被多模块使用的构建函数应归属 `core.services.poll` 或 `core.utils`，不得留在 `core.telegram`。
- **DRY 版本号规范**：版本号必须在且仅在 `core/__init__.py` 中定义，其他文件通过 `from core import __version__` 引用，禁止硬编码副本。CI 增加 `grep -r --exclude=core/__init__.py '"1\.[0-9]+\.[0-9]+"' core/` 检查。
- **条件兜底日志**：`if-elif-else` 中 `else` 分支采用兜底行为，必须添加 `logger.debug` 或注释说明。
- **全局状态访问显式依赖**：`get_xxx_repo()` 等函数必须显式接受依赖参数，禁止隐式依赖全局单例，返回前验证实例可用性。
- **依赖同步检查**：移除任何第三方库支持时，PR 必须同时更新 `requirements.txt` 或 `pyproject.toml`。
- **删除接口语义一致性**：Web API 与 Telegram 命令的删除类接口语义应一致（都按 ID 或都按条件），若不一致必须在 PR 描述中说明原因。

### 文档与流程
- 中英文 README，`wiki/` 沉淀架构与维护记录，重要变更记入 `CHANGELOG.md`。贡献流程见 `CONTRIBUTING.md`，PR 需通过 CI 检查。
- **增量审查规范**：优先采用增量审查，审查时主动扫描上下文相似模式并横向对比已知热点调用；比对 PR 描述与变更文件列表，检查关联资源是否同步；对修复类 PR 显式验证行为等价性。增加最小横向检查清单：变更函数的上层调用方、同模块下相同模式问题、上一轮未修复问题。功能移除类 PR 必须扩大边界检查至 `tests/`、`docs/`、`.github/`。
- **大型变更全量重审**：对于超过 2000 行或新增核心模块的 PR，强制执行一次全量重审，发现重复模式、跨文件集成和测试盲区。
- **docstring 一致性**：抽取或新增函数时，必须核对 docstring 与实际返回类型、副作用一致。异步方法 docstring 必须在前 10 个字符内包含“异步”或“Async”。
- **三联检查清单**：对于底栏/消息模板类修改，必须检查代码逻辑、i18n 字符串、注释/文档三处一致性。
- **历史遗留问题提醒**：增量审查中，若 PR 未涉及上一轮标记的“未修复问题”，审查报告末尾需增加“遗留风险提醒”段落，包含具体 Issue 编号。
- **提交信息交叉验证**：当 PR 描述中声明“修复 Issue A/B/C”时，审查者必须逐项确认相关文件是否在变更范围内，若不在需标记为“声明未实现”。
- **审查报告质量门禁**：审查报告中的“已修复”必须指向具体 commit hash。历史问题累计超过 3 次反思未修复则升级为架构债会议议题。
