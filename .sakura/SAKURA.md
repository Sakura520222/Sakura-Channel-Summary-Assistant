# Sakura-Bot 项目概述

## 项目简介
Sakura-Bot 是一款基于 AI 的智能 Telegram 频道管理工具，面向频道管理员提供自动内容总结、智能问答、投票生成与历史分析能力，帮助提升社群运营效率。

## 技术栈
- **语言**：Python 3.13+
- **交互**：Telegram Bot API
- **AI**：OpenAI 兼容 API（DeepSeek、OpenAI 等）+ RAG 检索增强生成 + 独立提示词系统
- **调度**：内置定时器（每日、每周等）
- **存储**：文件/轻量存储，对话总结持久化 + ChromaDB 向量存储
- **前端**：Vue 3 + TypeScript（向量存储管理界面）
- **测试**：pytest + 集成测试 + 覆盖率分析
- **质量**：Ruff、Black、pre-commit
- **部署**：Docker & Docker Compose，跨平台启动脚本
- **配置**：dotenv 环境变量 + JSON 配置文件，统一由 `core/config.py` 加载

## 架构设计与关键决策
项目按功能划分 `core/` 子包，AI、命令、配置、事件处理、转发等职责分离。

**关键决策与已知架构债务**：
- **命令元数据 + Handler 注册表模式**：声明式设计，超过 5 个工具的类应使用注册表字典映射。
- **配置双通道**：环境变量 + JSON 热更新并存，需统一布尔值解析工具。
- **延迟导入规范**：禁止在函数/方法内部使用 `import` 作为常规依赖管理，除非注释说明循环导入。
- **文件持久化**：直接读写 JSON 文件无锁机制，多 worker 下必须使用 `filelock` 或 Redis 分布式锁。
- **审计系统**：非阻塞写入 + 失败仅记录日志，符合可观测性最佳实践。
- **仓储模式反模式**：全局单例仓储 + 无依赖传递导致单元测试困难，应强制依赖注入重构。
- **版本号管理债务**：强制单点定义（仅 `core/__init__.py`）并加入 CI 检查。
- **参数清洗一致性**：空字符串转 `None` 应统一为工具函数 `sanitize_optional_str`。
- **双存储对称性要求**：平行存储结构必须提供对称的增删改查接口，合并读取时使用业务键去重。
- **向量存储 API**：支持分页、语义搜索、批量操作，注意路由注册顺序（精确路径必须在参数化路径之前）。
- **异步后台迁移**：迁移脚本必须在后台异步执行，启动时检查完成状态，未完成时禁用相关功能并持续提醒。
- **横切逻辑收拢**：权限检查等横切关注点应统一到专用模块，禁止新旧两套机制并存。
- **测试参数化覆盖**：遍历构建键盘、菜单等固定枚举范围时，测试必须使用 `@pytest.mark.parametrize` 覆盖全部常量，并反向验证回调路由存在性。

## 已知问题和注意事项
1. **配置布尔陷阱**：`bool('false')` 为 `True`，统一使用 `parse_bool`。
2. **文件 I/O 安全**：禁止全量读取大文件；API 响应仅返回文件名或相对路径。
3. **异常屏蔽**：`finally` 块中避免调用可能抛异常的清理函数。
4. **时区比较不一致**：写入 MySQL DATETIME 必须使用 naive UTC datetime，禁止 aware datetime 直接传入。
5. **空值保护遗漏**：调用外部 API 后需检查所有嵌套字段。
6. **异常静默处理**：禁止 `except` 块仅返回空值不记录日志；权限拒绝必须记录审计日志。
7. **深链接用户名空值**：构造 `https://t.me/` 链接需检查目标是否为空。
8. **parse_mode 与内容安全**：启用 Markdown 时动态内容必须转义，推荐 `parse_mode="MarkdownV2"`。
9. **空字符串 vs None 语义**：Web API 中需统一约定空字符串的语义。
10. **循环体内常量创建**：不可变常量必须提升到模块级。
11. **业务类型名字符串散落**：两个以上位置使用的字符串类型名必须改用 `Enum`。
12. **删除接口语义不一致**：WebUI 与 Telegram 接口语义需统一或在 PR 中说明原因。
13. **配置热重载非原子**：`load_config()` + `update_module_variables()` 两步之间无锁。
14. **数据库查询性能隐患**：模糊查询（`LIKE %xxx%`）需确认索引，新增查询必须评估数据量并添加 `LIMIT`。
15. **并发安全盲区**：模块级 `dict` 作为协程间共享状态时必须使用 `asyncio.Lock` 保护。
16. **批量操作无长度限制**：所有接受列表的 API 端点必须进行长度上限校验并返回 400。
17. **整数转换无异常保护**：从请求提取字符串转 `int` 时，必须使用 `try/except ValueError` 并返回 400。
18. **异步上下文阻塞 I/O**：异步函数中禁止直接使用 `open()`、`time.sleep()` 等阻塞调用，必须使用 `aiofiles` 或 `executor`。
19. **迁移脚本幂等性缺失**：所有迁移必须包含：幂等性标记、`--dry-run`、预估执行时间日志、回滚方案。
20. **分批循环无限风险**：`while True` 循环必须有明确的退出条件检查和最大迭代次数保护。
21. **sender_id 为 None 的处理**：Telegram 特殊事件（如频道帖子）中 `sender_id` 可能为 `None`，权限判断必须显式处理、记录警告日志并立即 `return False`，禁止仅打日志不返回。
22. **审计日志不完整**：所有权限拒绝、配置修改、异常捕获必须记录可追溯的日志。
23. **技术债 Issue 强制关联**：审查发现“不阻塞但重要的遗留问题”时，必须立即创建 Issue 并在 PR 描述中关联。

## 审查中发现的重要模式
- **声明式命令注册表**：审查重点在元数据与 handler 参数类型匹配。
- **防御性日志读取**：使用 `deque(maxlen=N)` 替代完整读取。
- **异步资源泄漏防范**：`nonlocal` 标志位 + `finally` 清理模式。
- **增量审查横向扫描**：必须检查变更函数的上层调用方、同模块相同模式问题、上一轮未修复问题。
- **显式白名单 + 默认降级**：防止 Markdown 解析崩溃。
- **常量提升缺失反模式**：函数循环体内定义不可变常量。
- **isawaitable 语义化**：用 `inspect.isawaitable` 替代 `hasattr` 检查。
- **敏感信息脱敏**：精确匹配 + 后缀匹配（`_token`、`_secret`）比前缀匹配更安全。
- **对象引用去重是反模式**：必须使用不可变业务键去重。
- **FastAPI 路由顺序陷阱**：精确路径（如 `/clear`）必须在参数化路径（如 `/{doc_id}`）之前注册。
- **“先取后删”分批模式风险**：高并发下需考虑 ChromaDB 行为，增加最大迭代次数保护。
- **辅助函数特化分支泄漏**：若辅助函数对特定类型保留手工分支，说明抽象层级不足。
- **碎片化权限检查模式**：大量 `if sender_id not in ADMIN_LIST: return` 重复代码，必须收敛到统一模块。
- **重构完成度验证**：宣称“抽取共享函数”的 PR，审查者必须全局搜索旧模式残留，强制创建技术债 Issue。
- **嵌套异常处理**：网络调用（如 `event.respond`）的外层兜底异常处理是必要的，但应避免过度嵌套。
- **边界日志必须伴随显式返回**：权限/状态检查中记录警告日志后，必须立即 `return` 或 `raise`。
- **增量修复的边界验证**：修复边界条件时，必须扫描该函数的所有直接调用方确认契约一致性。
- **None 的权限语义**：所有权限函数入口必须检查 `None` 并明确 fail-closed 行为。

## 团队约定和规范
### 代码与测试
- 强制使用 Black 格式化、Ruff 静态检查，pre-commit 自动执行。
- pytest 测试遵循 `test_*.py` 命名，新增端点必须包含：正常路径、异常路径、边界条件。
- 异步代码必须使用 `pytest-asyncio` 编写并发安全测试。
- 测试对实现细节耦合时须注释说明技术债，优先使用公共方法。
- 测试参数化强制：对于固定枚举范围的功能，测试必须使用 `@pytest.mark.parametrize` 覆盖全部枚举常量，并验证回调路由存在性。
- 修复类 PR 应包含针对原问题场景的测试用例（+1 行逻辑对应 +N 行测试）。

### 配置管理
- 敏感凭证通过 `.env` 注入，非敏感项使用 JSON；所有配置经 `core/config.py` 校验。
- 布尔解析强制使用项目级统一函数 `parse_bool`，禁止直接 `bool()`。
- 新增配置项必须全链路追踪类型转换与默认值一致性。
- 环境变量中的组件标识符统一使用小写，避免日志聚合失效。

### 文件 I/O 与安全规范
- 禁止全量读取未知大小文件；所有 `open()` 调用必须使用上下文管理器。
- API 响应禁止暴露服务器文件路径。
- 异步上下文中禁止直接使用同步 `open()`，必须使用 `aiofiles`。
- 共享文件 handler 添加前必须检查 `handler not in logger.handlers`，防止重复。

### 数据处理与防御
- 时区比较确保 naive/aware 一致；写入 MySQL DATETIME 必须通过统一转换函数。
- 调用外部 API 后必须在所有出现位置检查空值。
- 函数内部禁止定义不可变常量；两个及以上位置使用的字符串类型名必须改用 `Enum`。
- 整数转换必须捕获 `ValueError` 并返回 400。
- 批量操作必须强制加入长度上限校验；分批循环必须有最大迭代次数保护。
- 迁移脚本必须四要素齐全：幂等性、`--dry-run`、执行时间日志、回滚 SQL 注释。
- 权限判断中 `None` 值必须显式处理、记录警告日志并立即返回 `False`。
- 所有 `except` 块捕获异常后必须通过 `event.answer`、`event.respond` 或日志向用户反馈。

### 审查流程规范
- 增量审查强制检查：搜索本次修改函数的所有调用方；检查上轮警告是否修复；对提取的辅助函数搜索是否可复用。
- PR 描述-diff 一致性检查：审查者必须逐条核对 PR 描述中的变更点是否在 diff 中体现。
- 版本号变更强制检查：修改版本号是否仅在 `core/__init__.py` 中进行。
- 删除接口语义一致性：Web API 与 Telegram 命令语义应一致，不一致须在 PR 中说明。
- 警告抑制一致性：新增 `warnings.catch_warnings` 时，必须同步处理项目中相同警告类型。
- 重构完成度验证：宣称“抽取共享函数”时，审查者必须全局搜索旧模式残留点位，强制创建技术债 Issue。
- 工具函数归属原则：纯函数必须放在 `utils/` 层，禁止从基础设施模块导出业务无关工具。
- 调用方反向扫描强制：新增工具函数替代原有模式时，必须扫描所有潜在调用点。
