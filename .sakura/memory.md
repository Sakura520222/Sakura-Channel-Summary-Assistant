## 常见代码问题和审查要点
- 布尔配置陷阱：`bool('false') == True`，必须类型感知解析；自定义布尔解析函数须显式处理 None、int、str 类型，禁止使用 `bool(value)` 作为未经检查的兜底。
- 配置值转换分散重复：应抽取统一工具函数，确保初始化、热更新、持久化三环节类型转换一致。
- 读写对称性：修复读取侧须同步检查写入侧。
- 跨层依赖：`core.services` 从 `core.telegram` 反向引用标记架构债。
- 异常屏蔽：`finally` 裸调资源清理须包裹 `try/except` 并记日志。
- 构建后覆盖反模式：优先参数化。
- 时区混用：`datetime` 比较必须统一 `aware`，禁止静默兜底。
- LLM/API 响应空值保护：所有外部数据访问须逐级校验字段存在性，封装 `safe_get_nested` 工具函数。
- 异常静默处理：`except` 仅返回空值必须加 `logger.warning`。
- 延迟导入：非循环依赖一律置模块顶部；若必须函数内导入，放函数最前并注释原因，禁止夹在业务逻辑中。
- 闭包可读性：`run_in_executor` 逻辑超一行或涉及外部变量，强制使用具名局部函数。
- DRY 与实体标识符：提取 Telegram 实体用户名或 ID 模式出现 ≥2 次必须抽取为 `get_channel_display_identifier`。
- docstring 一致性：新抽取函数必须核对文档描述与返回类型、副作用；异步方法 docstring 前 10 字符内须含“异步”或“Async”。
- 不变性保护注释：前置守卫保证非 `None` 后使用，鼓励加注释。
- 字符串拼接空值过滤：`" ".join()` 与 `\n` 混用时，应对每个元素 `strip()` 或调整列表顺序避免多余空格；主动过滤 None/空字符串。
- 配置变量引用检查：直接使用模块级配置变量时，需确认热更新机制或添加守卫与注释。
- i18n 与代码解耦：i18n 中的 emoji 或特殊符号在代码层附加，或使用参数化模板。
- 深链接处理：涉及 `https://t.me/` 链接构造时，必须检查目标用户名/ID 是否为空，失败时记录 warning 并提前返回。
- 注释同步规则：注释中出现的具体输出格式（emoji、换行、链接），必须与代码实现保持一致，或改为引用变量名。
- 消息模板三联检查：消息构建、i18n 字符串、注释/文档中出现的格式必须一致，审查时三方对照。
- 白名单/黑名单合理性审查：遇到显式白名单，必须反向检查项目中是否存在“本应属于白名单但未包含”的合法调用。
- 兜底逻辑显式化：`else` 分支采用兜底行为时必须添加 `logger.debug` 或注释说明是设计选择。

## 近期审查模式总结
- 增量+quick 策略对补丁修复高效，需主动扫描提交信息识别非代码变更，并比对变更文件列表。
- 历史审查上下文让后续轮次精准定位，应持续维护。
- 配置新增/变更必须全链路追踪：环境变量→解析函数→模块变量→热更新→业务使用。
- 修复引入数据类型提升须主动追溯新的边缘情况。
- 增量审查中通过主动浏览文件上下文进行“横向扫描”，可捕获仅看 diff 遗漏的重复模式。
- 审查驱动修复模式趋向成熟：开发者能快速响应并一次性修复多个独立问题。
- 全量重审相比增量审查在发现系统性问题上优势明显，建议大型变更（超2000行）或新增核心模块时强制执行。
- 对涉及配置热更新、Token 管理的 Issue，自动触发环境变量一致性检查提示。
- 三合一检查清单：底栏/消息模板类修改必须检查代码逻辑、i18n 字符串、注释/文档三处一致性。
- 增量审查必须包含遗留风险提醒：若 PR 未涉及上一轮标记的“未修复问题”，报告末尾增加“遗留风险提醒”段落。
- 格式-解析模式一致性：消息内容中使用的格式符号必须与发送时的 `parse_mode` 匹配，不确定时注释声明预期模式。
- 版本号 DRY：版本号必须在且仅在 `core/__init__.py` 中定义，禁止硬编码副本。

## 规范建议和经验教训
- 建立统一布尔解析函数 `parse_bool(value, default)`，拒绝未经检查的 `bool(value)` 兜底。
- 重复逻辑出现 ≥2 次应抽取。
- 抽取公共函数须审视归属模块，避免层间依赖。
- 异步事件泄漏：`nonlocal` 标志位+`finally` 清理，确保标志位在注册前设置，无提前 `return` 绕过。
- 时区比较一致性规则：强制 UTC，无法统一则拒绝比较并记录错误。
- LLM 响应访问模式强制校验，考虑 lint 追踪。
- 延迟导入集中收敛，函数内延迟导入须置于函数体最前并附注释。
- 增量审查附带关联文件检查：变更涉及配置引用调整，须确认资源文件同步修改。
- 提交信息与实际变更范围校验：描述与 diff 不符时须人工确认。
- 重构验证必须显式说明行为等价性。
- 文档撰写必须即时：方法抽取后立即编写/修订 docstring。
- 微小可读性问题（缺注释）积累后维护成本高，值得指出。
- 新增数据访问层时，必须检查对应数据库迁移脚本是否存在且版本递增。
- API 调用（base64.decode、json.loads 等）必须被 try/except 包裹，日志级别至少 WARNING。
- 异步单例的 start/stop 必须使用 asyncio.Lock 保护，防止并发冲突。
- 缓存淘汰原则：使用 `OrderedDict` 实现 LRU 时，新增单条导致超限仅移除一条，避免不必要的循环。
- 历史问题闭环验证：审查报告中标记“已修复”的问题，必须引用具体 commit hash 或 diff 片段作为证据。
- 日期时间处理：对未知来源的 datetime 或字符串，统一使用 `parse_date` 工具函数封装类型判断和格式解析，拒绝直接 `[:10]` 截取。
- 类型守卫顺序：处理混合类型时优先检查明确类型（`isinstance`），再检查鸭子类型（`hasattr`），最后才是兜底。
