## 常见代码问题和审查要点
- 布尔配置陷阱：`bool('false') == True`，必须类型感知解析。
- 配置值转换分散重复：应抽取统一工具函数。
- 读写对称性：修复读取侧须同步检查写入侧。
- 跨层依赖：`core.services` 从 `core.telegram` 反向引用标记架构债。
- 异常屏蔽：`finally` 裸调资源清理须包裹 `try/except` 并记日志。
- 构建后覆盖反模式：优先参数化。
- 时区混用：`datetime` 比较必须统一 `aware`，禁止静默兜底。
- LLM/API 响应空值保护：一处加保护需全局扫描所有访问点。
- 异常静默处理：`except` 仅返回空值必须加 `logger.warning`。
- 延迟导入：非循环依赖一律置模块顶部；若必须函数内导入，放函数最前并注释原因，禁止夹在业务逻辑中。
- 闭包可读性：`run_in_executor` 的逻辑超过一行或涉及外部变量，强制使用具名局部函数。
- DRY 与实体标识符：提取 Telegram 实体用户名或 ID 的模式（`get_entity → hasattr(username) → str(id)`）出现 ≥2 次必须抽取为工具函数 `get_channel_display_identifier`。
- docstring 一致性：新抽取的函数/方法，必须核对文档描述与返回类型、副作用是否一致，重点关注大小写、空值。
- 不变性保护注释：局部变量经前置守卫保证非 `None` 后使用，鼓励加注释说明，提升可读性。

## 近期审查模式总结
- 增量+quick 策略对补丁修复高效，但需主动扫描提交信息识别非代码变更（如 prompt 清理），并比对变更文件列表，弥补 diff 盲区。
- 历史审查上下文让后续轮次精准定位，应持续维护。
- 配置新增/变更必须全链路追踪：环境变量→模块变量→热更新→持久化→业务使用。
- 修复引入数据类型提升须主动追溯新的边缘情况（如时区）。
- 增量审查中通过主动浏览文件上下文进行“横向扫描”，可捕获仅看 diff 遗漏的重复模式，应作为最佳实践。
- 审查驱动修复模式趋向成熟：开发者能快速响应并一次性修复多个独立问题，DRY 意识增强。
- 对改善级别修复（如断言改注释）保持务实容忍，只要不降低安全性或可维护性。

## 规范建议和经验教训
- 建立统一布尔解析函数 `parse_bool(value, default)`。
- 重复逻辑出现 ≥2 次应抽取。
- 抽取公共函数须审视归属模块，避免层间依赖。
- 异步事件泄漏：`nonlocal` 标志位+`finally` 清理，确保标志位在注册前设置，无提前 `return` 绕过。
- 时区比较一致性规则：强制 UTC，无法统一则拒绝比较并记录错误。
- LLM 响应访问模式强制校验，考虑 lint 追踪。
- 延迟导入集中收敛，函数内延迟导入必须置于函数体最前并附注释，严禁夹在业务逻辑中。
- 增量审查附带关联文件检查：变更若涉及删除/调整配置引用，须确认对应资源文件是否同步修改。
- 提交信息与实际变更范围校验：提交描述中提到而 diff 未体现的内容，须人工确认识别遗漏。
- 重构验证必须显式说明行为等价性，尤其是纯方法抽取。
- 文档撰写必须即时：方法抽取后立即编写/修订 docstring，审查者应将其作为重构一部分检查。
- 微小可读性问题（缺注释）积累后维护成本高，值得指出。
