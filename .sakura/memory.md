## 常见代码问题和审查要点
- 布尔配置陷阱：`bool('false') == True`，必须类型感知解析；自定义布尔解析函数须显式处理 None、int、str 类型，禁止使用 `bool(value)` 作为未经检查的兜底。
- 配置值转换分散重复：应抽取统一工具函数，确保初始化、热更新、持久化三环节类型转换一致。
- 读写对称性：修复读取侧须同步检查写入侧。
- 跨层依赖：`core.services` 从 `core.telegram` 反向引用标记架构债。
- 异常屏蔽：`finally` 裸调资源清理须包裹 `try/except` 并记日志。
- 时区混用：`datetime` 比较必须统一 `aware`，禁止静默兜底。
- LLM/API 响应空值保护：所有外部数据访问须逐级校验字段存在性，封装 `safe_get_nested` 工具函数。
- 异常静默处理：`except` 仅返回空值必须加 `logger.warning`。
- 延迟导入：非循环依赖一律置模块顶部；若必须函数内导入，放函数最前并注释原因，禁止夹在业务逻辑中。
- 闭包可读性：`run_in_executor` 逻辑超一行或涉及外部变量，强制使用具名局部函数。
- DRY 与实体标识符：提取 Telegram 实体用户名或 ID 模式出现 ≥2 次必须抽取为 `get_channel_display_identifier`。
- 常量提升规则：循环体内禁止定义 `frozenset`、`tuple`、`dict` 等不可变常量，必须提升至模块级别。
- 字符串类型名转枚举：业务类型名出现在 ≥2 个位置时，必须使用 `Enum` 或模块级常量统一管理。
- i18n 强制解耦：所有面向用户的字符串（日志、异常、通知）必须通过 `get_text` 调用，禁止硬编码。
- 版本号 DRY：版本号必须在且仅在 `core/__init__.py` 中定义，禁止其他文件硬编码。
- 依赖同步检查：移除数据库适配器等模块时，必须同步检查 `requirements.txt` / `pyproject.toml` 是否移除对应依赖。
- 白名单/黑名单合理性审查：使用显式白名单时须反向检查遗漏项；使用前缀匹配时须注释说明扩展预期。
- 兜底逻辑显式化：`else` 分支采用兜底行为时必须添加 `logger.debug` 或注释说明是设计选择。

## 近期审查模式总结
- 增量审查必须包含“遗留风险提醒”段落，主动关联历史未修复问题并附带 Issue 编号。
- 提交信息与实际变更范围必须交叉验证：PR 描述中“修复 #X”须逐项确认对应文件在本次 diff 中。
- 数据库适配器替换类 PR 必须扩大检查边界：依赖文件、全局搜索旧适配器名、迁移脚本版本链、CHANGELOG。
- 死代码残留检测：函数内联后须 `grep -r "原函数名"` 检查调用残留，建议 CI 集成 `vulture`。
- i18n 完整性检查：`grep -r "get_text"` 提取 key，再 `grep -r "key" locales/` 验证存在性；同时扫描未使用 `get_text` 的中文字符串。
- 常量提升建议需区分边界：若常量值未来可能依赖配置或动态计算，则不应强制提升。
- 版本号变更自动触发关联检查：README、CHANGELOG、deployment 脚本、硬编码副本。

## 规范建议和经验教训
- 移除功能类 PR 的审查盲区常在 diff 外：依赖文件、全局搜索、异常传播路径、版本号策略。
- 增量审查必须追踪修改行的下游依赖（至少 1 层），尤其是条件分支、白名单/黑名单、`parse_mode` 等控制字段。
- 审查报告标记“已修复”必须引用具体代码行号或 commit hash 作为证据，避免假闭环。
- 启动健康检查分散超过 2 处时，应建议提取为统一状态管理模块。
- 仓储模式反模式：全局单例仓储 + 无依赖传递应强制要求依赖注入重构。
- 日期时间处理：对未知来源的 datetime 或字符串，统一使用 `parse_date` 工具函数封装，拒绝直接 `[:10]` 截取。
- 类型守卫顺序：优先检查 `isinstance`，再检查 `hasattr`，最后兜底。
- 消息模板四联检查：代码逻辑 + i18n key 存在性 + i18n 内容 + 注释示例 → 四者一致。
