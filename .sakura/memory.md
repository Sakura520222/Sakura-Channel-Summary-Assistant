## 常见代码问题和审查要点
- 布尔配置陷阱：`bool('false') == True`，必须类型感知解析。
- 配置值转换分散重复：应抽取统一工具函数。
- 读写对称性：修复读取侧须同步检查写入侧。
- 跨层依赖：`core.services` 从 `core.telegram` 反向引用标记架构债。
- 异常屏蔽：`finally` 裸调资源清理须包裹 `try/except` 并记日志。
- 构建后覆盖反模式：优先参数化。
- 时区混用：`datetime` 比较必须统一 `aware`，禁止静默兜底。
- LLM/API 响应空值保护：一处加保护需全局扫描所有访问点。
- 异常静默处理：`except` 仅返回空值必须加 `logger.warning`。
- 延迟导入：非循环依赖一律置模块顶部；若必须函数内导入，放函数最前并注释原因，禁止夹在业务逻辑中。
- 闭包可读性：`run_in_executor` 的逻辑超过一行或涉及外部变量，强制使用具名局部函数。
- DRY 与实体标识符：提取 Telegram 实体用户名或 ID 的模式出现 ≥2 次必须抽取为 `get_channel_display_identifier`。
- docstring 一致性：新抽取函数必须核对文档描述与返回类型、副作用是否一致。
- 不变性保护注释：前置守卫保证非 `None` 后使用，鼓励加注释说明。
- 字符串拼接空值过滤：使用 `" ".join` 构建消息时，应主动过滤 `None`、空字符串项。
- 配置变量引用检查：直接使用模块级配置变量时，需确认热更新机制或添加守卫与注释。
- i18n 与代码解耦：i18n 中的 emoji 或特殊符号最好在代码层附加，或使用参数化模板。

## 近期审查模式总结
- 增量+quick 策略对补丁修复高效，但需主动扫描提交信息识别非代码变更，并比对变更文件列表。
- 历史审查上下文让后续轮次精准定位，应持续维护。
- 配置新增/变更必须全链路追踪：环境变量→模块变量→热更新→持久化→业务使用。
- 修复引入数据类型提升须主动追溯新的边缘情况。
- 增量审查中通过主动浏览文件上下文进行“横向扫描”，可捕获仅看 diff 遗漏的重复模式。
- 审查驱动修复模式趋向成熟：开发者能快速响应并一次性修复多个独立问题。
- 全量重审相比增量审查在发现系统性问题上具有明显优势，建议大型变更（超2000行）或新增核心模块时强制执行。
- 对涉及配置热更新、Token 管理的 Issue，应自动触发环境变量一致性检查提示。

## 规范建议和经验教训
- 建立统一布尔解析函数 `parse_bool(value, default)`。
- 重复逻辑出现 ≥2 次应抽取。
- 抽取公共函数须审视归属模块，避免层间依赖。
- 异步事件泄漏：`nonlocal` 标志位+`finally` 清理，确保标志位在注册前设置，无提前 `return` 绕过。
- 时区比较一致性规则：强制 UTC，无法统一则拒绝比较并记录错误。
- LLM 响应访问模式强制校验，考虑 lint 追踪。
- 延迟导入集中收敛，函数内延迟导入必须置于函数体最前并附注释。
- 增量审查附带关联文件检查：变更涉及配置引用调整，须确认资源文件同步修改。
- 提交信息与实际变更范围校验：描述与 diff 不符时须人工确认。
- 重构验证必须显式说明行为等价性。
- 文档撰写必须即时：方法抽取后立即编写/修订 docstring。
- 微小可读性问题（缺注释）积累后维护成本高，值得指出。
- 新增数据访问层时，必须检查对应数据库迁移脚本是否存在且版本递增。
- API 调用（base64.decode、json.loads 等）必须被 try/except 包裹，日志级别至少 WARNING。
- 异步单例的 start/stop 必须使用 asyncio.Lock 保护，防止并发冲突。
- Bug 优先级判定参考：功能完全失效→high；功能可用但行为错误→medium；功能可用但体验不佳→low。

## 需要特别关注的领域
- `core/config.py` 配置加载与热更新，布尔、数字等类型转换。
- `core.telegram` 与 `core.services` 导入方向与职责边界。
- 事件处理器生命周期管理，尤其是 shutdown 钩子完整性。
- 配置持久化写入侧类型一致性。
- `datetime` 对象时区处理。
- 外部 API 响应防御性访问。
- 异常处理中没有日志的静默兜底。
- Telegram 实体标识符获取的碎片化重复，需强制统一。
- 延迟导入的规范化摆放。
- 函数/方法 docstring 与实现的一致性。
- 异步后台任务的异常捕获与处理（asyncio.create_task 必须配合 await 或 add_done_callback）。
- 简单缓存的数据结构（如 dict）必须考虑 TTL 和最大容量，防止内存泄漏。
