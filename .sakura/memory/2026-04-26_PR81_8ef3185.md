### 深度反思

#### 1. 审查质量评估
- **覆盖度**：审查覆盖了死代码、空值保护、重复实现、弃用 API、异步阻塞、日期比较缺陷、流式/非流式路径不一致等关键问题，对变更的六个文件均有针对性检出，整体覆盖面广。细微遗漏在于未检查 `vector_store.py` 中 `search_similar` 的返回值契约是否随本次改动引入新的不一致，但影响很小。
- **准确度**：严重等级判定基本合理。`agent_loop.py` 完全未集成为 **critical** 是准确的；流式响应空值保护不足列为 **critical** 也恰当。`_TOOL_NAME_MAP` 未使用标记为 **major** 略高，实际属于 `minor` 死代码。主要评论中存在一条 **critical** 重复（`qa_engine_v3.py:736` 与 `:737` 实为同一问题），轻微误判在于将同一缺陷拆成两条，但未影响实质判断。
- **完整性**：审查充分理解了变更意图（引入 Agentic RAG），并揭示出架构过渡阶段的割裂：新模块未接入、流式/非流式双轨行为不一致。不仅指出局部缺陷，还把握了整体架构债务，上下文理解深入。

#### 2. 发现的模式
- **代码模式**：
  - **重复工具函数**：`_message_to_dict` 在 `agent_loop.py` 和 `qa_engine_v3.py` 一字不差地复现，且历史审查曾强调“重复逻辑≥2次应抽取”，说明团队仍缺乏自动化检测或遵循度不足。
  - **死代码倾向**：`agent_loop.py` 整个文件未被引用，`_TOOL_NAME_MAP` 无引用，反映在快速迭代中缺乏集成测试或导入追踪。
  - **异步阻塞隐患**：`_execute_rerank` 同步调用 `httpx.Client` 直接阻塞事件循环，项目中已有同类问题但未被修正，可能成为并发瓶颈。
  - **字符串日期比较**：使用字典序比较日期，忽略时区和 ISO 格式变体，是典型的语义不匹配。
- **架构观察**：
  - **双轨演进不彻底**：项目同时存在流式（Agentic RAG）和非流式（固定流水线）两条路径，此次更新仅替换了流式路径，遗留了行为不一致和技术债务，后续统一检索逻辑势在必行。
  - **职责边界模糊**：Agent 循环逻辑被内联到 QA 引擎内部，而非使用独立的 `agent_loop.py` 模块，导致架构预期（独立循环控制）与现实（内联实现）脱节。

#### 3. 规范完善建议
- **新增审查规则**：
  - **死代码检查**：对所有新增模块，必须验证其是否被 `import` 且在至少一条执行路径上可达，否则标记为 `critical`。
  - **异步阻塞检查**：任何 `async` 上下文中调用可能阻塞的 I/O 操作（如 HTTP、文件读写）必须包装进 `run_in_executor` 或使用异步版本，否则标记为 `major`。
  - **日期比较类型安全**：涉及日期/时间的比较必须解析为 `datetime` 对象并统一时区，禁止字符串字典序比较。
  - **弃用 API 检查**：扫描 `asyncio.get_event_loop()`、`loop.run_until_complete` 等在目标 Python 版本中已弃用的调用，建议替换。
- **项目特有规范**：
  - **双路径同步**：由于存在 `process_query_stream`（流式）和 `_handle_content_query_v3`（非流式），任何对检索或 Agent 逻辑的变更都需同时保证两条路径行为一致，审查时需显式检查。
  - **重复消除自动化**：推荐在 CI 中加入相似代码检测（如 jscpd），对于跨文件重复自动告警。
  - **Agent 工具模块治理**：`agent_tools.py` 应作为 Agent 所有工具实现的唯一出口，禁止在引擎文件中内联工具调用，以强化模块边界。

#### 4. 经验教训
- **集成验证是死代码的第一道防线**：审查者应养成习惯，对新文件搜索项目级引用，避免“僵尸模块”混入主干。
- **流式中断补偿**：流式生成时已发出部分内容后异常，降级路径无法撤销已发数据，审查必须关注这种不可逆场景，建议要求至少发送补救消息或设置“部分回答已失效”标记。
- **历史审查上下文复用**：项目记忆中已有“重复抽取”和“异常屏蔽”等教训，本次审查再次遇到同类问题，说明应将历史审查发现的模式固化为 lint 规则或 PR 模板检查项，减少人工遗漏。
- **架构迁移需整体完成**：当引入重大模式转变（如固定流水线→Agentic），应避免分步提交导致中间状态不一致。审查时应要求作者在 PR 描述中明确说明路径差异以及后续整合计划，否则按完整性不足处理。

这些反思强化了静态分析与人工审查的结合，并为后续迭代提供了具体的规则补充和流程改进方向。
