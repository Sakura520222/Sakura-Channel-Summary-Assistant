## 2. 发现的模式

### 代码模式
- **“集合白名单”模式**：本次变更将隐式规则改为显式枚举，提升了确定性。项目中类似模式（如支持的 `parse_mode` 类型列表）应该统一使用 `frozenset` 或 `Enum`。
- **模块级常量提升缺失**：多次出现常量定义在函数循环体内的模式，反映开发者对 Python 对象生命周期和性能优化细节的关注度不足。**应作为常见反模式加入审查清单**。
- **硬编码类型名**：直接写入字符串字面量 `"submission_approved"` 等，缺少集中定义（如 `NotificationType` 枚举）。项目中若有多个地方引用同一类型名，存在不同步风险。

### 架构观察
- **处理器专用逻辑**：`mainbot_push_handler.py` 中的通知类型判断逻辑是独立的，未与通知类型的定义中心（如 `models/notification.py`）耦合。这降低了变更的影响面，但也增加了类型名散落在各处的维护成本。
- **历史债务积累**：`qa_engine_v3.py:920` 的日期类型兜底问题已持续多轮未修复，说明该类文件属于“低频变更但高风险”区域，建议定期全量审查或专项重构。

## 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规则：

### 新增规则
1. **常量定义位置规则**  
   - 若一个变量是不依赖于实例或函数参数的不变常量，必须在模块级别定义。  
   - 函数/方法内部禁止定义 `frozenset`、`tuple`、`dict` 等不可变常量（除非该常量依赖于运行时动态计算的值）。  
   - **标签**：`performance`、`maintainability`

2. **字符串类型名转枚举规则**  
   - 当业务类型名（如 `"submission_approved"`）出现在两个或以上不同位置时，必须使用 `Enum` 或模块级常量统一管理。  
   - 集合匹配时必须引用枚举成员，禁止硬编码字符串字面量。  
   - **标签**：`DRY`、`type-safety`

3. **迭代中不可变容器创建检查**  
   - 在 `for` 或 `while` 循环体中，如果创建了集合、字典等容器且每次内容相同，应提示提升到循环外部。  
   - **标签**：`performance`

### 强化现有规则
- **白名单/黑名单合理性审查**：当使用显式白名单时，除了检查“应属于但未包含”的条目，还应该检查“未来可能新增的类型是否会被遗漏”，并建议添加注释或防御性兜底。本次审查已间接覆盖，但可明确写入规则示例。

## 4. 经验教训

### 值得在未来审查中关注的要点
- **增量审查必须追踪修改行的“上下游依赖”**  
  本次变更只改了 `parse_mode` 的判断条件，但未检查该变量如何被下游使用（例如 `send_message` 中是否依赖 `parse_mode='HTML'` 之外的默认行为）。建议增量审查中加入“调用链快速浏览”步骤（至少 1 层）。
- **对“常量提升”类建议要区分边界**  
  虽然本次建议是正确的，但需注意：如果常量的值未来可能依赖配置或动态计算，则提升为模块级反而会引入隐式依赖。审查时应判断常量是否“真·不变”。
- **历史遗留问题的“所有权”标记**  
  `qa_engine_v3.py` 的问题多轮未修，可能是因为没有明确的责任人。审查报告可以建议团队为每个“遗留风险”分配一个 Issue Owner。

### 对特定技术/模式的审查经验
- **`frozenset` 用于成员测试**：这是一个好的模式，比 `tuple` 或 `list` 更语义化且可哈希。审查时遇到集合匹配优先推荐 `frozenset`。
- **`startswith` 前缀匹配的陷阱**：曾作为黑名单用于排除某些类型，但当类型数量增多且前缀不稳定时容易出错。本次改为显式枚举是正确的方向。日后可总结为“前缀匹配仅适用于日志、监控等非业务关键路径”。
