## 3. 规范完善建议

### 建议新增/修改的审查规则

| 规则ID | 名称 | 说明 |
|--------|------|------|
| **SR-011** | 列名安全校验强制 | 任何拼接列名的 SQL 生成点，必须通过静态白名单或元数据查询结果校验，禁止直接使用 `dict.keys()` |
| **SR-012** | 批量审计日志强制 | 同一 PR 内新增多个写操作端点时，必须统一检查审计日志覆盖率，不得遗漏任一写路径 |
| **SR-013** | 前端动态渲染内容转义 | WebUI 中展示来自数据库的用户内容时，必须使用 `v-html` 禁止或手动转义 |
| **SR-014** | 重启/清空类操作并发锁 | 影响全局状态的路由（重启、清空全部数据）必须持有 `asyncio.Lock` 并记录锁等待时间 |

### 项目特有规范强化

- **工具函数归属原则** 应上升到 **强制 Lint 检查**：在 CI 中扫描重复函数定义（相似度 > 80% 且跨文件）。
- **审计日志示例化**：在 `docs/audit-examples.md` 中提供 CRUD、清空表、重启等场景的标准日志调用模板。
- **N+1 查询检测**：建议在 `mysql.py` 层封装 `batch_count_tables()` 方法，供 `list_tables` 使用，避免审查中反复指正。

## 4. 经验教训

### 值得在未来审查中关注的要点

- **白名单校验的完整性**：看到 `table in ALLOWED` 后，必须继续检查列名、排序字段等是否也受控。
- **重复工具函数的隐性成本**：一旦发现两个路由模块有相同辅助函数，应立刻要求抽取到 `deps.py` 或 `utils/`。
- **“审计日志”关键词扫描**：在 diff 中搜索 `POST|PUT|DELETE` 路由，必须同步搜索 `record_audit` 调用。缺失即 major。
- **前端 JSON 数据渲染安全**：审查 `v-html` 或 `{{ data.value }}` 的直接展示，评估是否可能引入 XSS。

### 对特定技术/模式的审查经验

| 模式 | 审查经验 |
|------|----------|
| **动态 SQL 列名** | 永远假设用户控制 `data.keys()`。即使表名白名单，列名也必须白名单。最佳实践：`set_clause = ", ".join(f"`{col}`=%s" for col in safe_cols)` |
| **数据库管理 UI** | 重点关注：列名注入、错误回显是否泄露结构、COUNT 性能、事务完整性。 |
| **FastAPI + 异步 DB** | 检查每个 `await db.execute` 前是否可能阻塞（如同步驱动）。优先使用 `asyncmy` 或 `aiomysql`。 |
| **前端路由与后端一致性** | 发现新增前端页面时，必须确认对应后端 API 权限与审计到位，往往后端是短板。 |
