## 2. 发现的模式

### 代码模式
- **“命令元数据 + Handler 注册表”模式**：集中描述命令参数、类型、是否需要确认，再通过 `OPERATION_HANDLERS` 映射到具体函数。这是良好的**声明式设计**，易于扩展。
- **参数提取函数 `_require_param` / `_param`**：频繁使用字符串键名访问参数，缺乏类型安全，导致出现了 `type: ignore`。
- **配置访问直接调用 `get_config()` / `write_config()`**：无配置层抽象，事务性更新困难，容易出现部分写入成功部分失败的情况。

### 架构观察
- **Web API 层与 Bot 命令层共享业务逻辑**：`commands.py` 中的 `_op_*` 函数同时被 Web 和 Telegram 使用，这是合理的复用。但该文件已 749 行，未来会膨胀。
- **文件作为持久化存储**：`schedules.py` 中直接读写 JSON 文件，无数据库或锁机制，成为并发风险点。
- **历史遗留规范存在但未严格执行**：项目有 `SAKURA.md` 明确定义了版本号 DRY，但本次 PR 仍然写了 6 处硬编码，说明**规范检查未自动化**。

## 3. 规范完善建议

基于本次审查，建议对项目规范或审查规则做以下补充：

### 新增审查规则
| 规则 ID | 描述 |
|--------|------|
| `rule-dry-002` | 版本号硬编码检测：除 `core/__init__.py` 外，禁止任何文件出现 `"X.Y.Z"` 形式的版本字符串（除非是注释或迁移脚本且有显式免责注释）。 |
| `rule-api-001` | Web API 与 Telegram 命令的删除类接口语义应一致（都按 ID 或都按条件），若不一致必须在 PR 描述中说明原因。 |
| `rule-error-003` | 从请求中提取整数参数时，必须使用 `try/except ValueError` 捕获类型转换异常，返回 400 而非 500。 |
| `rule-concurrency-001` | 对文件系统进行“检查-读-改-写”操作时，禁止使用 `exists()` 预检查，必须使用 `try/except` 捕获 `FileNotFoundError`，并评估是否需要文件锁。 |
| `rule-import-003` | 标准库模块（如 `asyncio`, `json`, `os`）的导入必须置于文件顶部，禁止函数内延迟导入，除非有循环依赖（须注释）。 |

### 项目特有的规范要点提炼
- **版本号 DRY** 已经是显式规则，应加入 CI 检查（`grep -r --exclude=core/__init__.py '"1\.[0-9]+\.[0-9]+"' core/`）。
- **危险操作二次确认** 已被良好实现，可推广到更多修改类操作。
- **审计日志** 已存在，建议标准化日志字段格式。

## 4. 经验教训

### 值得在未来审查中关注的要点
1. **PR 描述与代码实际变更的一致性**：本次 PR 描述提到了 `core/__init__.py` 版本号更新，但审查报告未确认该文件是否真的在 diff 中（虽然列表里有它）。未来应主动交叉验证。
2. **新人/贡献者是否了解项目规范**：6 处版本号硬编码暗示开发者可能不知道 `SAKURA.md`。审查时应建议项目增加 **PR 模板** 或在贡献指南中突出此类规范。
3. **文件操作并发问题**：在缺少数据库的项目中，频繁出现文件读写竞态。审查时应主动扫描所有 `exists()` + `open()` 模式，标记为潜在风险。
4. **类型系统的边界**：`type: ignore` 往往代表类型设计不完整，审查时应追踪其原因，判断是否可以改进类型注解。
5. **迁移脚本的版本号策略**：迁移脚本内硬编码版本号用于“记录引入迁移的版本”，这种做法本身是合理的，但应与业务版本号解耦（例如使用独立的时间戳或序列号）。

### 对特定技术或模式的审查经验
- **声明式命令注册表**：审查重点应放在 **元数据与实际 handler 的参数类型是否匹配**，以及 **options/choices 是否与 handler 内部的校验逻辑重复**。
- **文件锁的必要性评估**：若项目是单用户/低并发场景，可接受无锁；若 WebUI 支持多标签页且操作频繁，需要加锁。本次 PR 中的 `schedules.py` 操作频率低，可标记为 minor，但应提示未来风险。
- **错误类型的分层**：`list_userbot_channels` 中内层返回 `{success: False}`、外层抛 `HTTPException` 的模式不一致，审查建议“统一策略”是正确的，但未指明最佳实践——内层应直接抛 `HTTPException` 或返回 None 由外层转换。
