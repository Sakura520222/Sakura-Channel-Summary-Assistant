- **代码模式**：
    1.  **“碎片化权限检查”模式**：项目中存在大量的 `if sender_id not in ADMIN_LIST: return` 模式的重复代码，这是典型的横切关注点（cross-cutting concern）泄露到业务逻辑中。本次重构试图收敛，但仅覆盖了冰山一角。
    2.  **防御性编程不足**：`sender_id is None` 导致误判，暴露了开发者默认假设所有事件都有 `sender_id` 的思维。在处理 Telegram 事件时，应对 `channel post`、`service message` 等特殊类型保持警惕。
    3.  **硬编码字符串的蜕变**：帮助文本从“硬编码命令名”改为“元组+动态生成”，这是一个从反模式到健康模式的积极转变。

### 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规则：

- **规则草案（新增）**：**重构完成度验证规则**

- **强化规则**：**空值安全与假阳性/假阴性审查**
    - **要点**：对于涉及权限、状态判断的逻辑，审查者必须检查 `None`、`0`、空字符串等“边界真值”的处理。尤其要关注 `None not in LIST` 在 Python 中的行为（返回 `True`，可能导致逻辑反转）。必须要求开发者显式处理 `None` 分支并添加日志。

- **新增建议**：**测试参数化覆盖规则**
    - **建议**：对于遍历构建键盘、菜单等具有固定枚举范围的功能，单元测试应使用 `@pytest.mark.parametrize` 或循环遍历所有已定义的枚举常量，而不是硬编码部分值。审查时应指出此类不完整的测试。

### 4. 经验教训

- **审查要点**：
    1.  **重构 PR 必扫调用方**：当看到新增 `auth.py` 时，必须立即执行 `git grep "ADMIN_LIST"` 或类似搜索，评估影响范围。这是一种高效发现潜在债务的技巧。
    2.  **关注“静默失败”倾向**：`errors='ignore'` 被改为 `backslashreplace`，说明之前存在“将编码错误静默丢弃”的习惯。需要警惕项目中是否还有其他 `ignore` 行为。
    3.  **环境变量默认值的大小写敏感**：日志组件名称的大小写不一致看似微小，但会导致日志聚合、过滤系统失效。这提示项目应统一约定：所有组件标识符使用小写。

- **技术/模式经验**：
    - Telethon 的 `event.answer()` 和 `event.respond()` 都是网络调用，异常处理应分两层：一层保证主流程不崩溃，一层保证用户能收到反馈。外层兜底异常处理是必要的。
    - Telegram Callback Data 的 64 字节限制是一个硬性平台约束，`build_callback` 的编码后检查 + 测试覆盖是一个**优秀的防御模式**，值得在其他与外部 API 交互的地方推广（如长度、频率、格式限制）。
