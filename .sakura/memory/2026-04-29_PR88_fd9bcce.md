## 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规则：

### 新增规则
1. **迁移文件检查**：新增数据访问层（repo）时，必须检查是否存在对应的数据库迁移脚本（如 Alembic），且版本号递增。
2. **API 调用异常处理统一检查**：所有 `base64.b64decode`、`json.loads`、`requests.get`（或 `aiohttp`）等可能抛出异常的操作，必须被 `try/except` 包裹，且错误日志级别至少为 WARNING。
3. **异步单例的 start/stop 必须加锁**：全局后台任务管理器的生命周期方法必须使用 `asyncio.Lock` 保护，防止并发调用导致资源重复创建或无法正确关闭。

### 强化已有规则
- **延迟导入位置**：从“建议”升级为“强制”规则，除非存在明确循环依赖（需注释说明），否则所有 import 必须位于模块顶部。
- **国际化硬编码**：所有直接返回给用户的字符串（尤其是错误消息）必须通过 `get_text` 获取，禁止任何语言硬编码。自动化 lint 可考虑扫描中文字符串。

## 4. 经验教训

### 未来审查关注点
- **关闭钩子完整性**：当新增 `shutdown` 或 `close` 方法时，必须反查调用链，确认是否被主流程（如 `main.py`、`atexit`、signal handler）调用。
- **缓存数据结构**：使用 `dict` 做简单缓存时，必须同时考虑 TTL 和最大容量，防止内存泄漏。`functools.lru_cache` 或 `cachetools.TTLCache` 是更好选择。
- **参数命名与使用一致性**：方法签名中的参数如果未使用，应移除或添加 `_` 前缀（如 `_client`）明确表示未使用，并加注释原因。

### 对特定技术的审查经验
- **asyncio 后台任务**：审查时需确认每个 `asyncio.create_task` 都有对应的 `await` 或 `add_done_callback` 处理异常，否则未捕获异常会静默消失。
- **配置文件热更新**：当多个地方同一配置项赋值时，应怀疑是否有人为覆盖风险，建议统一配置读取入口（如 `get_config()` 函数）。
