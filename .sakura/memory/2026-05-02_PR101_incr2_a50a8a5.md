## 发现的模式

### 代码模式
1. **手动维护派生状态**：`on_config_updated` 中同时更新 `_config` 和 `_source_channel_ids`，而非通过 `set_config` 统一管理 —— 这是 **“状态同步脆弱模式”**。
2. **重复工具函数**：`_extract_source_channel_ids` 在两个类中各自实现 —— 提示可提取为 `channel_utils.extract_id_from_url()`。
3. **测试中过度手动清理**：使用 `NamedTemporaryFile` 后显式 `unlink`，在跨平台场景下不安全 —— 反映项目测试基础设施对 `tmp_path` 的使用不够普及。

### 架构观察
- **配置热重载链路清晰**：`ConfigManager` → 事件 → `ForwardingHandler.on_config_updated`，职责分离较好。
- **技术债分层明确**：同步 I/O 问题被标记为 `SR-009`，且允许在非阻塞 PR 中延期，说明项目已建立“可接受的技术债管理机制”。
- **增量更改谨慎**：将 `_get_current_source_channels` 从实时解析改为读缓存，消除了原性能瓶颈，但要求初始化时确保缓存已填充 —— 这是典型的“预计算 + 缓存一致性”模式。

## 经验教训

### 值得在未来审查中关注的要点
1. **“setter 绕过”是一个经常被忽视的 bug 来源**：尤其当 setter 中包含校验、日志、缓存刷新或副作用时。审查时应扫描所有 `self._xxx = value` 赋值，确认是否存在对应的 `set_xxx` 方法。
2. **增量审查的盲区**：本次 PR 中 `forwarding_initializer.py` 仅修改了 1 行（`+1/-1`），但审查者仍然发现了其内部调用自身 `_extract_source_channels` 的逻辑重复 —— 这提示审查者不仅要看 diff，还要看调用链上下文。
3. **异常捕获过窄的长期风险**：对 Telethon 特定 `ValueError` 的依赖在版本升级时可能断裂。建议在注释中写明“依据 Telethon v1.x 行为”，并考虑捕获更通用的 `Exception` 后通过 `isinstance` 判断。

### 对特定技术的审查经验
- **Telethon 转发链路**：重点关注 `forward_messages` 异常类型、`from_peer` 解析失败场景、实体缓存过期处理。这些是消息机器人的常见生产故障点。
- **配置热重载与 watchdog**：必须测试“文件内容没变但 mtime 变了”的场景（如 `touch`），否则可能引入无限重载循环。
