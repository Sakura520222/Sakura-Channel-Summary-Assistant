### 2. 发现的模式

#### 代码模式

**1. DRY 修复的“系统性根除”模式**
- 统一从 `core` 导入 `__version__`，无局部 fallback
- 迁移脚本的版本号与业务版本**解耦**（仅注释同步）
- **可推广规范**：所有全局常量（如默认超时、路径前缀）应单一定义

**2. “优先新语义 + 兼容旧语义”的 API 演进模式**
- 转发规则删除：先检查频道对（新），再回退到 rule_index（旧）
- 未使用 feature flag 或版本头，依赖参数存在性判断
- **风险**：若新语义与旧语义在边界条件重叠（如频道名为空字符串），可能混淆

**3. 文件 I/O 的安全模式**
```python
async with lock:
    try:
        data = await _read_file()
    except FileNotFoundError:
        return  # 提前退出，不记录成功日志
    # 修改 data
    await _write_file(data)
```
- **特点**：`FileNotFoundError` 作为“正常缺失”处理，不抛异常
- **优点**：避免 `exists()` 竞态
- **缺点**：三层 try-async with-try 嵌套，可读性下降

**4. 敏感信息脱敏的演进**
- 旧：`"key" in key.lower()`（误匹配 "keyboard"）
- 新：精确匹配 + 后缀匹配（`_token`、`_secret`、`_password`）
- **观察**：后缀匹配比前缀匹配更安全（`api_token` 而非 `token_abc`）

#### 架构观察

**1. 配置与常量的分层**
- `core/__init__.py`：业务版本号
- `core/config.py`：运行时配置
- 模块级常量（如 `LOG_LEVEL_OPTIONS`）：路由层专用
- **依赖方向**：路由层 → 模块常量 → 无外部依赖（合理）

**2. 异步锁的使用边界**
- `asyncio.Lock` 保护 JSON 文件的“读-改-写”
- **隐含假设**：单进程单事件循环（FastAPI 默认）
- **架构风险**：若未来多 worker（gunicorn -w 4），锁失效，文件可能损坏

**3. 命令中心的扩展性设计**
- `COMMAND_METADATA` + `OPERATION_HANDLERS` 分离
- `_default_metadata` 为未知命令提供默认值
- **优点**：新增命令只需注册，无需改路由
- **缺点**：未知命令的默认风险等级需谨慎（当前未确认）

### 4. 经验教训

#### 值得在未来审查中关注的要点

**1. 异步文件锁的作用域幻觉**
- **教训**：审查者容易默认 `asyncio.Lock` 能跨请求保护文件，但多 worker 下失效
- **未来检查点**：审查文件操作时，主动询问“是否可能多进程部署？”

**2. “修复历史问题”的增量审查陷阱**
- **教训**：PR 声称“修复版本号硬编码”，审查者验证了 6 处，但可能遗漏动态生成的版本号（如 f-string 拼接）
- **未来检查点**：使用正则搜索 `r"\b1\.[0-9]+\.[0-9]+\b"`，而非仅搜索 `"1.8.4"`

**3. 前端变更的“副作用”常被后端审查忽略**
- **本次案例**：前端 `loadSummaryTimes` 从 `Promise.all` 移出，后端审查者只关注“是否合理”，未验证前端错误处理是否依赖原有并行语义
- **未来检查点**：涉及前端异步时序变更时，要求 PR 描述说明错误处理路径的变化

**4. 迁移脚本的版本号注释容易混淆**
- **本次案例**：`migrate_forwarding_table_v1.py` 中的版本号注释改为 `v1.8.4`，但迁移脚本本身与业务版本无关
- **教训**：迁移脚本应使用时间戳或独立序列号（如 `v20250101`），避免与业务版本耦合
- **规范建议**：废止迁移脚本中的业务版本号引用

#### 对特定技术/模式的审查经验

**`asyncio.Lock` + 提前 return 模式**
- **模式**：
  ```python
  async with lock:
      try:
          data = await read()
      except FileNotFoundError:
          return  # 在锁内提前退出
      # 写回文件
  ```
- **审查要点**：
  - 提前 return 是否释放锁？（`async with` 会自动释放 ✅）
  - 提前 return 是否跳过必要的清理？（无 ✅）
  - 是否应记录日志？（取决于业务语义 ✅）

**“参数为空字符串” vs “参数缺失”的歧义**
- **FastAPI 行为**：`params.get("key")` 对缺失字段返回 `None`，对空字符串返回 `""`
- **常见混淆**：前端可能传 `source_channel: ""` 表示“清空”，后端可能误解为“忽略”
- **审查经验**：遇到 `if source or target` 判断时，要确认空字符串的业务语义

**❌ 未涉及（非本次 PR 范围）：**

- 测试覆盖缺失（项目记忆中强调但本次未验证）

#### 增量变更是否引入新风险？

**潜在风险 1：** 按频道对删除的优先级高于 `rule_index`，如果前端错误地将频道对字段设为空字符串，会回退到 `rule_index` 删除，可能误删规则
- **缓解**：审查者已建议三分支结构（`if source and target / elif source or target / else`），可避免静默回退

**潜在风险 2：** `_default_metadata` 对未知命令的风险等级未确认，可能导致危险命令被误判为安全而跳过确认弹窗
- **缓解**：审查者已标注为 [suggestion]，要求确认默认风险等级

**潜在风险 3：** 文件 I/O 重构中，`_remove_file_if_exists` 使用 `try/except` 替代 `exists()`，但未检查文件删除后的竞态（如删除后立即被其他协程重建）
- **实际影响**：低，因为删除操作在锁内执行

#### 审查覆盖度是否因只看增量部分而下降？

**下降点 1：未验证依赖同步**
- 新增了 `commands` 路由，但未检查 `core/web_api/app.py` 是否已导入（描述中提到已注册，但 diff 可能不包含该文件）

**下降点 2：未检查类型注解的连锁影响**
- `CommandResultData = dict[str, Any]` 类型别名新增，但未检查使用处是否因类型收窄而产生新错误

**下降点 3：前端变更的后向兼容性**
- `SchedulesView.vue` 中 `loadSummaryTimes` 改为顺序执行，但未验证旧版本浏览器是否因异步时序变化而出现 UI 闪烁

**改进建议：** 增量审查应要求 PR 描述中包含“影响范围”章节，明确列出：
- 直接修改的文件
- 间接影响的文件（如被调用方、配置引用）
- 需要回归测试的功能点
