### 2. 发现的模式

#### 代码模式
- **辅助函数提取模式**：将重复的删除逻辑提取为 `_delete_*_by_*` 形式的私有函数，职责清晰
- **分批循环模式**：`while True: batch = collection.get(...); if not batch['ids']: break; delete(batch['ids'])` — 标准的“先取后删”模式
- **异常处理模式**：在 `_delete_document_by_collection` 中对 `int(doc_id)` 做了 `ValueError` 捕获，但其他端点仍存在未捕获的情况（不一致）

#### 架构观察
- **向量存储层抽象薄弱**：`vector_store.py` 直接依赖 ChromaDB 的具体 API（`get`, `delete` 方法签名），缺乏统一的数据访问接口
- **配置常量散落**：`CLEAR_COLLECTION_BATCH_SIZE` 和 `MAX_BATCH_DELETE_IDS` 定义在模块中部，未归集到 `config.py` 或常量区域
- **审计日志缺失**：本轮变更涉及删除操作，但审查未要求验证审计日志记录（项目记忆明确要求“所有写操作必须记录审计”）

### 4. 经验教训

#### 未来审查关注点
- **分批删除的事务性**：若中途失败，已删除的 ID 无法回滚，是否接受部分删除？应在 API 文档中明确
- **ChromaDB 的 `get(include=[])` 行为**：返回的 `batch['ids']` 是否一定是字符串列表？建议审查时标注依赖版本
- **循环边界测试**：当集合恰为 `BATCH_SIZE` 的整数倍时，最后一轮 `get` 是否可能返回空？当前代码 `if not batch['ids']: break` 可正确处理

#### 对特定技术/模式的审查经验
- **ChromaDB 批量删除**：ID 匹配是字符串精确匹配，无需转换类型，但需注意数字 ID 在 JSON 传输时可能被反序列化为整数，需在 API 层统一转字符串
- **“先取后删”模式的风险**：
  - 高并发下，取出的 ID 可能在删除前已被其他请求删除，`delete(ids=[...])` 对不存在的 ID 应静默忽略（ChromaDB 行为需验证）
  - 循环删除过程中新插入的文档不会被本轮删除，符合预期（删除的是“调用时刻的快照”）
