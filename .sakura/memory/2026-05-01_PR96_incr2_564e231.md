## 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规则：

| 规则 | 说明 |
|------|------|
| **去重键显式性检查** | 任何基于 `id(obj)` 或对象引用的去重，必须要求使用不可变业务键（如 `id`、`name`）或明确注释契约（“同一对象引用”）。 |
| **静态方法调用约定** | 若方法为 `@staticmethod`，应通过类名调用；若通过 `self` 调用，应改为实例方法或模块级函数。新增规范：“禁止静态方法通过 `self` 伪装调用”。 |
| **工具与引擎重复逻辑扫描** | 当新增一个工具函数或引擎方法时，必须 grep 项目中是否存在相似逻辑（如链接提取、频道解析）。 |
| **测试覆盖脆弱契约** | 对于依赖隐式契约的代码（如共享引用），测试必须包括**反例**：模拟契约被破坏时的行为（例如通过深拷贝）。 |

## 4. 经验教训

### 值得未来关注的要点
- **对象引用去重是反模式**：在序列化、深拷贝、多线程环境下极易失效。未来审查中应直接要求改为基于字段去重。
- **历史问题追踪不能只做“确认修复”**：还应该评估修复方案本身是否引入新风险（本次去重正是典型）。
- **增量审查的盲区**：容易忽略对“未变更但被变更函数所依赖的代码”的检查。必须强制做“调用方反向扫描”。

### 特定技术/模式的审查经验
- **双存储结构**：必须审查增删改查的对称性，以及合并读取时的去重策略是否鲁棒。
- **帮助消息与解析模式**：`Markdown` → `HTML` 迁移后，应全局检查是否有残留的 `parse_mode="Markdown"`。
