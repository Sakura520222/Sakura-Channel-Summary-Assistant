### 2. 发现的模式

#### 代码模式
| 模式 | 表现 | 评价 |
|------|------|------|
| **延迟导入过泛** | `system.py` 中 ≥7 处函数内 `import` | 符合项目约定（避免循环依赖），但已演变为“滥用”，建议集中到 `_get_db` 等统一获取函数 |
| **审计记录模式** | 成功路径总是 `success=True`，失败时未记录 | 模式设计不完整，应强制包裹 try/except，失败时也写入审计 |
| **常量重复定义** | 版本号 `"1.8.3"` 硬编码 | 违反项目 DRY 规范（已在项目记忆中记录） |

#### 架构观察
- ✅ **延迟导入模式解决循环依赖**：`core/handlers/__init__.py` 使用 `__getattr__` 懒加载，是优秀的架构债还债方案。
- ✅ **审计系统设计**：非阻塞写入 + 失败仅记录日志，符合运维可观测性最佳实践。
- ⚠️ **仓储模式缺失**：数据库操作仍直接调用 `db.cleanup_xxx`，未抽象仓储层，违反“仓储模式反模式”项目记忆中的建议。
- ⚠️ **配置热更新非原子**：`load_config()` + `update_module_variables()` 两步之间无锁，符合项目记忆中“配置一致性”常见问题。

### 3. 规范完善建议

基于本次审查，建议**新增或强化**以下审查规则：

#### 3.1 文件 I/O 安全规范（新增）
```markdown
- **禁止全量读取未知大小文件**：如 `Path.read_text()`、`file.read()` 必须限制读取行数/字节数，或使用流式/尾部读取。
- **API 响应禁止暴露服务器文件路径**：日志、配置等文件路径须脱敏（仅返回文件名或相对路径）。
```

#### 3.2 审计日志完整性规范（新增）
```markdown
- **所有写操作（创建/更新/删除）必须记录审计**：且 try/except 包裹，失败时 `success=False` 也必须写入审计。
- **审计记录必须包含操作前后的关键数据指纹**：如删除行数、清理天数等，便于回滚。
```

#### 3.3 配置与常量 DRY 规范（强化）
```markdown
- 版本号、默认超时时间、日志轮转大小等跨文件常量，必须定义在 `core/constants.py` 或 `core/__init__.py`，禁止硬编码。
- 配置默认值（如 `CleanupRequest.days=30`）在代码中只定义一次，其他引用处必须导入该常量。
```

#### 3.4 并发安全声明规范（新增）
```markdown
- 非原子性操作（如配置重载、缓存刷新）必须添加 `asyncio.Lock` 或 在文档中明确“不支持并发调用”，并在 API 响应头中添加 `X-Idempotency-Key` 建议。
```

#### 3.5 测试覆盖强制检查（强化）
```markdown
- 新增 API 端点必须包含：正常路径、异常路径（参数非法、依赖失败）、边界条件（空 body、超大输入）。
- 删除/重构函数后必须全局搜索调用残留（`grep -r "原函数名"`）。
```

### 4. 经验教训

#### 值得在未来审查中关注的要点
1. **日志文件读取端点**：必须审查是否存在 OOM 风险（尤其生产环境日志 GB 级）。
2. **清理类 API**：必须审查是否有事务边界（数据库操作失败时是否回滚？是否记录失败审计？）。
3. **前缀/字符串匹配逻辑**：必须检查匹配顺序（长前缀优先）和边界输入（含 `www.`、尾部斜杠、Unicode）。
4. **版本号一致性**：每次发布前必须全局搜索硬编码版本号（`grep -r '"1\.' --include="*.py"`）。
5. **前端状态管理**：检查是否有多个组件重复初始化同一 `localStorage`/`sessionStorage` 键。

#### 对特定技术/模式的审查经验
- **`asyncio` 协程检测**：`hasattr(obj, "__await__")` 不如 `inspect.isawaitable()` 准确，且后者兼容 `asyncio.Future` 和协程。
- **`Path.read_text()` 陷阱**：看似简洁，但生产环境大文件下是定时炸弹，必须要求使用 `aiofiles` + 逐行读取或 `tail -n` 方式。
- **审计日志模式**：审查时需确认所有写操作（尤其是管理端批量删除/清理）是否都有审计，且失败路径覆盖完整。

### 5. 手动全量重审反思

#### 是否发现了之前增量审查遗漏的问题？
**是**，本次全量重审暴露了**增量审查的典型盲区**：

| 遗漏类型 | 具体问题 | 增量审查为何遗漏 |
|---------|---------|----------------|
| **跨文件副作用** | `App.vue` 和 `AppLayout.vue` 重复初始化主题 | 增量聚焦单个文件变更，未看到两个文件同时修改 localStorage |
| **配置默认值冲突** | `request_data=None` 默认 90 天 vs Schema 默认 30 天 | 增量审查只看到 `cleanup_audit_logs_endpoint` 修改，未对比其他两个清理端点的一致性 |
| **依赖引入缺失** | 建议使用 `aiofiles` 但未检查是否已添加依赖 | 增量审查未触发“新增依赖检查”规则 |
| **测试缺口** | `request_data=None` 分支未覆盖 | 新增测试文件（`test_web_api_system_ops.py`）未覆盖边界条件 |
| **历史技术债关联** | `normalize_channel_id` 前缀歧义（项目记忆中已有类似问题） | 增量审查未主动关联项目记忆中的“白名单/黑名单合理性审查”规则 |

#### 全量重审的价值
- **发现隐藏的耦合**：主题初始化分散两个组件、清理端点行为不一致，只有全量扫描才能发现。
- **验证架构一致性**：全量重审可以检查新增代码是否遵守项目记忆中的规范（如版本号 DRY、审计完整性）。
- **边界条件覆盖**：增量审查容易忽略“空 body 输入”“超大文件输入”等边界场景。

#### 对审查流程的建议
- **强制增量审查 + 定期全量快照**：每次 PR 增量审查 + 每 5 个 PR 触发一次全量重审（或变更超过 30 个文件时触发）。
- **增量审查必须附带“受影响模块清单”**：列出本次修改可能影响的其他文件（如修改 `config.py` 须检查所有导入配置的模块）。
- **项目记忆主动关联**：审查工具应自动检索项目记忆中与本次变更路径相关的历史规则（如修改 `config.py` 时自动提示“布尔配置陷阱”规则）。
