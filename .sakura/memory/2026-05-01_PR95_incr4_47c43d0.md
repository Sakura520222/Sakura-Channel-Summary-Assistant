### 2. 发现的模式

#### 代码模式
- **防御性参数清洗**：`params.get("key") or None` 模式反复出现，用于将空字符串转为 `None`。项目应统一规范为工具函数 `sanitize_optional_str`。
- **两段式验证结构**：先验证“两者都有”，再验证“只有一个”，最后隐式走“两者都无”分支。这是清晰的替代嵌套 `if not A or not B` 的模式。
- **辅助函数提取降低嵌套**：将文件读写逻辑提取为独立函数，有效降低嵌套深度（三层→两层）。

#### 架构观察
- **文件作为简易存储**：`schedules.py` 使用 JSON 文件存储数据，未引入数据库。这在低频操作下可接受，但并发控制弱（仅单进程锁）。
- **Web API 与 Telegram 命令分离**：`commands.py` 同时服务于两类入口，但未抽象公共验证层（如参数清洗逻辑重复）。
- **配置/常量硬编码倾向**：`LAST_SUMMARY_FILE` 在模块级硬编码，虽未违反 DRY，但缺乏集中配置管理。

### 3. 规范完善建议

#### 新增审查规则
1. **参数清洗一致性检查**  
   - 规则：若多处出现 `param.get("key") or None`，应建议提取为 `get_optional_str(params, key)`。
   - 级别：建议

2. **两段式验证注释规范**  
   - 规则：`if A and B` 后紧跟 `if A or B` 的结构，必须添加注释说明“只提供一个参数”的意图。
   - 级别：建议（但可提升为必须若团队多次踩坑）

3. **文件路径脱敏检查增强**  
   - 规则：所有返回 JSON 中若包含 `path`、`file_path`、`dir` 等键，须检查值是否为绝对路径或包含服务器敏感信息。
   - 级别：严重（若暴露绝对路径）→ 警告（若仅暴露相对路径但可推断结构）

#### 项目特有规范要点
- **“增量审查遗留风险提醒”应标准化**：建议在审查模板中增加“已知未修复问题”区块，要求 reviewer 主动检查上轮标记的 `⚠️ 警告` 是否在本轮修复。
- **辅助函数复用性检查**：提取的辅助函数若仅在一个端点使用，应在审查中建议推广到其他相似端点（如 `list`/`update`）。

### 4. 经验教训

#### 值得关注的要点
- **空字符串 vs `None` 语义**：Web API 中空字符串应视为“未提供”还是“清空操作”？本次变更选择前者，但项目缺乏统一约定。建议在 PR 模板中增加“参数空值语义说明”复选框。
- **增量审查的盲区——调用方一致性**：本次变更只改了 Web API 路由，但未检查 Telegram 命令中对应的删除逻辑是否也需要同步重构。即使不在 diff 范围内，也应提示“建议检查其他入口的一致性”。
- **文件 I/O 异常处理传播**：`_read_summary_time_file` 不捕获 `FileNotFoundError` 的设计是合理的（让调用方决定降级策略），但应在函数文档中明确说明“可能抛出 `FileNotFoundError`”。

#### 对特定技术的审查经验
- **`asyncio.Lock` 的多进程限制**：审查时应强制要求注释说明部署假设（单 worker），并建议未来切换分布式锁的替换点。
- **`or None` 转换的副作用**：审查时需验证输入值的合法范围（如空字符串、`false` 字符串、`0` 数字），避免误转换。
