### 2. 发现的模式

#### 代码模式
- **过度使用延迟导入**：`vector_store.py` 中 `get_vector_store` 被重复延迟导入 7 次，说明开发者可能出于“避免循环导入”或“优化启动时间”的误解而滥用此模式，而项目规范已明确指出这已成为反模式。
- **重复验证逻辑**：`collection_name` 白名单验证在 5 个端点中重复，表明缺乏模块级常量和验证函数的抽象习惯。
- **防御性编程不一致**：有的地方进行严格的异常捕获（`try/except`），而有的地方（如 `update.message.text.strip()`）则假设输入一定有效，说明团队在防御层级上缺乏统一标准。
- **FastAPI 路由意识薄弱**：对路径匹配顺序（精确路径 vs 参数化路径）的理解不足，是常见的初学者错误。

#### 架构观察
- **向量存储 API 设计成熟**：虽然存在实现细节缺陷，但 API 整体设计（分页、语义搜索、批量操作、资源存在性检查）显示了对 RESTful 和实用性良好理解。
- **Telegram 键盘设计**：将键盘抽离为独立模块 (`keyboards.py`) 并定义为常量，是一个优秀的关注点分离，说明正在朝可维护的聊天机器人架构演进。
- **前端与后端解耦良好**：`VectorStoreView.vue` 独立管理向量存储的前端状态和 API 调用，没有混杂在通用组件中。

### 3. 规范完善建议

基于本次审查，建议向项目记忆添加以下审查规则：

#### 新增审查规则

| 规则 ID | 名称 | 说明 | 触发条件 |
|---------|------|------|----------|
| **SR-001** | FastAPI 路由注册顺序审查 | 检查是否存在 `/{param}/action` 注册在 `/{param}` 之前或之后导致的冲突。精确路径（如 `/clear`）必须在参数化路径（如 `/{doc_id}`）**之前**注册。 | 任何新增的 FastAPI 路由，如果存在两个路径有包含关系且包含路径为参数化。 |
| **SR-002** | 批量操作长度限制强制 | 所有接受列表（如 `doc_ids`）的 API 端点必须对其进行长度上限校验（如 `len(list) > MAX_LIMIT`），并返回 400 错误。 | 新增 `POST/DELETE` 接口，请求体包含 `list[]` 类型字段。 |
| **SR-003** | 延迟导入审查强化 | 禁止在函数/方法内部使用 `import` 作为常规依赖管理手段。除非有明确注释说明循环导入场景，否则必须提升到模块顶部。 | 审查时发现 `def` 或 `lambda` 内部出现 `import` 语句。 |
| **SR-004** | 字符串转整数前置校验 | 从请求路径或查询参数提取的字符串，在转换为 `int` 时必须使用 `try/except ValueError`，并返回 400 错误和明确提示。 | 调用 `int(str_value)` 的代码没有被 `try-except ValueError` 包围。 |

#### 项目特有规范要点强化
- **布尔配置陷阱**已记录，但本次未出现，仍需保持警惕。
- **参数空值语义**：在 `vector_store.py` 的搜索端点中，`collection` 参数空值的语义（搜索所有集合 vs 错误）应明确文档化，本次审查已指出缺失验证。

### 4. 经验教训

#### 值得在未来审查中关注的要点
1. **FastAPI 路由顺序**：这是一个反复出现的陷阱。未来审查中遇到 `/{param}` 和 `/{param}/sub-path` 并存的情况，必须**强制要求**单元测试覆盖两种请求路径，证明路由解析正确。
2. **匿名功能的副作用**：匿名投稿功能需额外关注：
   - 审计日志中是否记录了真实提交者？（应记录审核后台可见）。
   - 用户被禁言或封禁时，匿名投稿能否追溯？
   - 匿名投稿的数据导出（如合规要求）如何处理？
   - 本次审查已经关注了署名 fallback 问题，但对审计和追溯的覆盖可以更深。
3. **向量存储的容量规划**：审查中指出了批量删除无限制，但没有涉及向量存储整体容量上限（如最大文档数、最大集合大小）。未来的 PR 如果扩展向量存储，应要求提供容量规划说明。
4. **前端静默异常**：`Vue` 组件中的空 `catch` 块违反了项目规范，说明前端审查需要更严格地对照规范清单进行。建议将“禁止静默 catch”纳入 PR 模板的自检清单。

#### 对特定技术/模式的审查经验
- **ChromaDB 集成审查**：审查 `vector_store.py` 时，应额外关注：
  - ChromaDB 客户端的连接池管理（是否每次请求创建新客户端？是否有超时设置？）。
  - 向量维度的一致性（`get_document` 中拉取全量向量仅获取维度，是否有更轻量的元数据 API？）。
  - 对于 `collection.delete(ids=doc_ids)` 的大批量删除，是否需要分批（比如每 1000 条一次）以避免 ChromaDB 内部超时？
- **ConversationHandler 审查**：Telegram 的 `ConversationHandler` 是状态机，审查时应关注：
  - 所有状态是否有超时或取消机制（用户中途退出会不会残留状态）。
  - 键盘的 `persistent` 设置是否与状态机生命周期匹配（本次审查已指出潜在问题）。
  - 多个对话（如投稿和审核）是否可能并发冲突（需要锁或会话隔离）。
