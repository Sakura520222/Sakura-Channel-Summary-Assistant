  - **版本号硬编码（应为 minor）**：标记为 suggestion 过低，违反项目明确规范，应至少为 **minor**。

### 二、发现的模式

#### 代码模式观察
1. **降级逻辑清除不彻底**：原 SQLite 回退常伴随 `except` 空捕获，移除后暴露真实异常，但缺少全局异常处理审计。
2. **死代码残留模式**：方法内联后忘记删除原方法，表明缺乏 IDE 静态检查或 lint 规则（如 `pyflakes` 未启用）。
3. **i18n 与 code 耦合倾向**：开发者习惯直接写字符串而非调用 `get_text`，需要更强的 Code Review 门禁。

#### 架构观察
1. **依赖注入不完整**：`get_submission_repo()` 未传入 `db_manager`，暗示仓储层可能静态持有连接，是潜在的测试和并发隐患。
2. **启动检查逻辑分散**：`startup_notifier`、`mainbot_request_handler`、`qa_bot` 三处都有“数据库是否就绪”的检查，缺乏统一状态管理模块。
3. **版本号管理债务**：多处硬编码已形成模式，需通过 lint 规则强制单点定义。

### 三、规范完善建议

#### 新增审查规则

| 规则 ID | 规则描述 | 触发条件 | 建议标记 |
|---------|---------|---------|---------|
| **DB-DEP-01** | 移除数据库适配器时，必须检查 `requirements.txt` / `Pipfile` / `pyproject.toml` 是否同步移除依赖 | 变更文件中包含数据库模块删除 | **major** |
| **I18N-04** | 所有面向用户的字符串（日志、异常、通知）必须通过 `get_text` 调用，禁止硬编码 | 非注释字符串包含中文/英文且不在 i18n 函数内 | **major** |
| **ARCH-07** | 全局状态访问函数（如 `get_xxx_repo`）必须显式接受依赖参数，禁止隐式依赖 | 函数返回值构造未传入必需的连接/管理器参数 | **minor** |
| **VERSION-02** | Lint 规则：禁止在 `core/__init__.py` 以外硬编码 `__version__` | 其他文件出现 `"1.x.x"` 模式 | **minor** |

#### 修改现有规则
- **“消息模板三联检查”** 扩展为 **“用户可见字符串四联检查”**：代码逻辑 + i18n key 存在性 + i18n 内容 + 注释示例 → 四者一致。
- **“兜底逻辑显式化”** 补充：当 `else` 分支仅剩单一实现时，应添加 `assert isinstance(db, MySQLManager)` 或 `if not isinstance(...): raise`，而非静默兜底。

### 四、经验教训

#### 未来审查关注要点
1. **数据库适配器替换类 PR**：
   - 必须检查依赖文件（`requirements.txt`、`setup.py`、`Pipfile`）
   - 必须全局搜索旧适配器名称（正则 `sqlite|aiosqlite`）
   - 必须验证迁移脚本版本链连续性
   - 必须确认 CHANGELOG + 升级指南

2. **函数内联后的残留清理**：
   - 手动执行 `grep -r "原函数名" src/` 检查调用残留
   - 要求启用 IDE 的“未使用符号”高亮或 CI 集成 `vulture`

3. **i18n 完整性检查技巧**：
   - 审查时 `grep -r "get_text"` 列出所有 key，再 `grep -r "key" locales/` 验证存在
   - 检查未使用 `get_text` 的中文字符串（`grep -P "[\u4e00-\u9fff]"`）

#### 具体技术模式审查经验
- **仓储模式反模式**：全局单例仓储 + 无依赖传递 = 单元测试噩梦。遇到此类 PR 应强制要求依赖注入重构。
- **启动健康检查分散**：超过 2 处检查同一状态时，应建议提取为 `DatabaseHealthService`。
- **Breaking Change 标记缺失**：移除 SQLite 是大版本变更（需 2.0.0），但 PR 只到 1.8.2，审查时应质疑语义化版本合规性。

#### 审查工具改进建议
- 引入自定义脚本：在审查前自动运行 `grep -r "sqlite" --include="*.py"` 并输出报告
- 集成 `pylint` 的 `W0611`（未使用导入）和 `W0612`（未使用变量）可发现死代码
- 对大型重构类 PR 强制要求提供“清理清单”复选框（依赖移除、死代码删除、i18n 同步、文档更新）

### 总结：本次审查的核心启示

**审查者的盲区往往不在 diff 内，而在 diff 的“未提及之处”** —— 依赖文件、全局搜索、异常传播路径、版本号策略。对于“删除功能”类 PR，必须扩大检查边界至：
1. 构建系统依赖
2. 用户文档（README、CHANGELOG、迁移指南）
3. 测试中的硬编码断言
4. 其他模块的隐式导入

**评分 8/10 略显宽松**，扣除的 2 分应覆盖 i18n 破坏（-1）+ 版本号违规（-0.5）+ 未检查依赖文件（-0.5）。建议严格执行规范评分卡，避免“人情分”。
