## 3. 规范完善建议

基于本次审查，建议新增或强化以下规则：

### 新增规则 SR-013：异常分层与日志级别规范

> 在业务逻辑层捕获异常时，必须区分：
> - **预期异常**（如参数校验失败、资源不存在）：应使用 `WARNING` 或 `INFO` 级别，并直接 `raise` 到上层，不记录 error 日志。
> - **非预期异常**（如数据库连接失败、IO 错误）：应使用 `ERROR` 级别。
> 
> **检查方式**：审查者在增量代码中搜索 `except Exception:`，要求内部先捕获具体子类或使用 `except (ValueError, KeyError):` 这样的前置处理。

### 强化规则 SR-014：类型标注与实际行为一致性

> 所有公共函数（尤其是返回类型变更时）必须同步更新类型标注。若函数存在“有时返回 None，有时返回 int”的真实语义，应显式保留 `None` 分支而不强制转换。
> 
> **检查方式**：对比函数的 `return` 语句与类型标注，确保不存在“标注允许 None 但实际永远不返回 None”或相反的情况。

### 新增建议：数据库元数据缓存约定

> 对于 `describe_table`、`list_tables` 等读取 `INFORMATION_SCHEMA` 的操作，建议使用 `functools.lru_cache(maxsize=128)` 或 TTL 缓存（如 `cachetools`），并监听表结构变更事件（如 DDL 迁移后主动清除缓存）。
> 
> **检查方式**：若发现写入路径中调用 `describe_table` 且该表在单次请求中被多次查询，应要求添加缓存。

### 追加 PEP 8 导入检查

> 同一模块的多个导入必须合并，禁止拆分为多个独立 `import ... as` 块。可使用 `isort` 工具自动检查。

## 4. 经验教训

### 值得未来关注的要点

1. **修复 PR 的副作用检查**：当修复一个“返回 None 导致调用方出错”的问题时，必须同时检查：
   - 是否有调用方依赖 `None` 做特殊逻辑？
   - 新的返回值（如 `0`）是否会被误解为有效 ID？
   
   本次 `insert_row` 返回 `0` 而不是 `None`，可能导致调用方误以为插入成功并尝试用 `id=0` 去查询。

2. **审计日志的完整性**：不能只检查“是否调用了审计函数”，还要抽样确认**记录的内容是否足够重建操作**（如更新前的字段值）。

3. **增量审查的边界**：本次审查做得很好——没有被“9 个问题都修了”迷惑，而是深入检查了修复代码本身。这是增量审查的最佳实践。

### 对特定技术/模式的审查经验

- **Python `except Exception` 的审查要点**：要求开发者明确列出预期处理的异常类型，否则可能吞没 `KeyboardInterrupt`、`SystemExit` 或掩盖真正的 bug。
- **`lastrowid` 的使用陷阱**：MySQL 中对于没有自增主键的表，`lastrowid` 返回 `0`，不应将其包装成有效 ID。更安全的做法是：如果表定义了主键且为自增，则返回新 ID；否则返回 `None` 或抛出异常告知不支持。
