## 经验教训

### 值得未来关注的要点
1. **配置与格式的解耦**：当消息内容来自多个来源（用户输入、数据库、外部 API）时，不能假设内容安全。**优先使用纯文本**，仅在确认内容无特殊字符时才启用 Markdown。
2. **增量审查的盲区**：本次变更未包含 `qa_engine_v3.py`，但历史问题（`created_at` 类型兜底）仍在。**审查报告应主动提醒后续 PR 连带修复**，而不是仅标记“未修复”。
3. **三元表达式的复杂度阈值**：当条件超过两个分支，或分支逻辑涉及外部依赖时，应 **强制要求提取函数**。项目可设定：`if-elif-else` 嵌套深度 ≥2 或分支 ≥3 必须重构。

### 对特定技术/模式的审查经验
- **Telegram parse_mode**：审查时**必须交叉检查**：
  - 消息内容中的特殊字符（`_`、`*`、`[`、`(` 等）是否被转义（MarkdownV2 要求转义）
  - `parse_mode` 是否为 `None` 但内容中包含格式符号
  - `parse_mode = "HTML"` 时是否错误使用了 Markdown 语法（反之亦然）
- **版本号配置**：每次版本号变更 PR，审查员应**主动搜索** `git grep "version\|VERSION"` 找出所有硬编码位置。
