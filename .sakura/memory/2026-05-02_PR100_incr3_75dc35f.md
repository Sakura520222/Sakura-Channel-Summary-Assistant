## 4. 经验教训

### 值得在未来审查中关注的要点
- **返回值语义变更比类型变更更危险**：类型从 `int` 改 `int|None` 是显式化，但 `0 → None` 可能破坏 `if row_id:` 这类隐式假值判断。
- **增量审查的“调用方盲区”反复出现**：即使变更极小，也应触发对“谁在用这个函数”的快速思考。
- **异常透传模式容易被忽略**：许多审查能发现 `except Exception` 太宽，但很少注意到 `except ValueError: raise` 也会透传非预期的 `ValueError`。

### 对特定技术/模式的审查经验
- **`cursor.lastrowid` 在异步 MySQL 驱动中的行为**：应确认 `aiomysql` 下多语句执行时的行为是否与同步 `mysql-connector` 一致。本次未深入，但可作为未来抽象层审查的检查点。
- **泛型 `TypeVar` 的正确使用**：`maybe_await` 的修改是一个优秀范例，可推广为工具函数类型标注的标准写法。
