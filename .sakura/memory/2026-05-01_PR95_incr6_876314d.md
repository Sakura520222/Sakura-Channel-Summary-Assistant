### 2. 发现的模式

**代码模式**：
- **精准抑制模式**：使用 `warnings.catch_warnings()` + 上下文限定 + 精确匹配 `category` 和 `message`，这是 Python 警告处理的最佳实践
- **显式声明默认参数**：将 PTB 的隐式默认值 `per_message=False` 显式写出，提高可读性并防止框架版本行为变更
- **防御性注释**：详细说明为什么必须使用该模式，降低未来维护者误删或误改的风险

**架构观察**：
- PTB 框架使用深度较高（会话管理、回调查询），项目对框架内部警告机制有明确的处理策略（不盲目全局禁用）
- 增量审查历史中反复出现 **“已知技术债未被修复”** 的模式：`asyncio.Lock` 跨进程风险已连续多轮标记，但始终未进入本次变更范围。这表明项目可能存在“高优先级问题被长期搁置”的风险积累

### 3. 规范完善建议

**新增审查规则**：

1. **警告抑制一致性检查**：
   - 当 PR 中新增 `warnings.catch_warnings()` 时，必须 grep 项目其他位置是否有相同警告类型发生但未抑制的情况

2. **依赖版本范围关联分析**：

### 4. 经验教训

**值得关注的要点**：
- **警告抑制的脆弱性**：字符串模式匹配（即使使用通配符）在框架升级时可能静默失效。防御性做法：除注释外，可添加单元测试验证警告是否被抑制（例如 mock `warnings.warn` 并断言未调用）
- **增量审查的盲区再发现**：本次再次验证——增量审查容易遗漏“同一个警告在其他文件出现”的问题。建议在工具层面实现**跨文件模式搜索**作为审查前置步骤
- **历史技术债的“麻木效应”**：连续多轮标记同一问题（如 `asyncio.Lock`）但未修复，可能导致审查人员对该问题逐渐“视而不见”。需要机制强制关联：例如在 PR 描述中必须回答“是否涉及已知遗留风险文件？”

**技术/模式审查经验**：
- **PTB 会话管理**：`ConversationHandler` 与 `CallbackQueryHandler` 搭配 `per_message=False` 是合法但易触发警告的模式。经验结论——PTB 设计上更倾向于 `per_message=True` 追踪消息，但在纯按钮驱动的会话中，`per_message=False` 是正确的。未来类似场景应统一使用本次的抑制模式
- **异步锁的跨进程问题**：模块级 `asyncio.Lock` 在多 worker（gunicorn + uvicorn workers）下完全失效。审查经验——一旦看到 `asyncio.Lock` 保护文件操作，必须强制确认部署拓扑。如果是多进程，必须要求更换为 `filelock` 或 Redis 分布式锁
