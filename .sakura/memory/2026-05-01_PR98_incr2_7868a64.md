### 2. 发现的模式

#### 代码模式
- **“局部修复 + 扩散依赖”模式**：抽取 `utc_now_naive()` 后，仅替换了 1 处调用点，却期望其他 24 处“未来迁移”。这实质是**将技术债转移给未来**，而非解决。
- **迁移脚本的“一次性思维”**：未设计幂等性、未支持回滚、未记录执行时间，表明开发者将迁移视为“运行即弃”的一次性脚本，而非生产数据变更工具。
- **Shell 脚本的防御性不足**：`start.sh` 中使用 `(cd web && npm install)` 的子 shell 隔离是对的，但未检查 `npm` 命令是否存在，也未处理 `package-lock.json` 不一致时的 `npm ci` 降级。

#### 架构观察
- **工具函数职责混乱**：`mysql.py` 既管理连接池，又导出时区工具函数，导致 `submission_repo` 被迫依赖数据库底层模块。**抽象层泄露**已成为系统性模式。
- **时区处理的“非侵入式妥协”**：选择在应用层做 `replace(tzinfo=None)` 而非在数据库驱动层统一配置，导致 24+ 处手动转换点无法收敛。这是典型的**架构决策债务**。

### 4. 经验教训

#### 值得关注的审查要点
1. **异步函数中的子进程调用**：`asyncio.create_subprocess_exec` 本身不阻塞，但调用方必须 `await proc.wait()`，否则协程泄漏。审查时应强制检查返回值处理。
2. **迁移脚本的 `fromisoformat` 陷阱**：Python 的 `datetime.fromisoformat` 保留时区信息，但 `strftime` 会丢弃时区，必须显式 `astimezone(UTC)`。
3. **批量提交的边界条件**：当 `commit` 同时出现在循环内和循环外时，必须检查是否为整数倍触发的重复提交。

#### 对特定技术的审查经验
- **Shell 脚本中的 npm 命令**：永远不要假设 `npm install` 后 `node_modules/.bin` 中的工具可用，因为 `--omit=dev` 会跳过构建工具。必须使用 `npx` 或检查 `npm list`。
- **MySQL DATETIME 与 Python datetime**：`aiomysql` 默认将 DATETIME 列返回为 `datetime.datetime` 对象，若该对象带 `tzinfo`，直接写回会导致 MySQL 报错。必须在连接层设置 `conv` 转换器。

#### 增量变更引入的新风险
- **依赖方向反转**：`submission_repo` 现在依赖 `mysql.py` 的工具函数，若后续将 `mysql.py` 拆分为多个文件，会导致广泛的导入变更。
- **幂等性假象**：迁移脚本的 `LIKE '%T%'` 查询在修复后确实不会重复处理，但若脚本中途失败重跑，已修复的记录不会重复修复，但未提交的事务会丢失。缺少 `ALTER TABLE` 添加 `migrated_at` 列的设计。

#### 覆盖度下降的根本原因
1. **只看增量差异，忽略调用传递**：审查者发现 `utc_now_naive()` 新增，但未扫描 `mysql.py` 中其他 Repo 是否也使用了 `datetime.now(UTC)`。
2. **依赖“开发者自述”而非代码验证**：PR 描述声称“修复 reviewed_at 时区问题”，审查者未实际检查 `update_submission` 方法。
3. **缺少自动化辅助**：未使用 `grep` 统计 `datetime.now(UTC)` 的实际出现次数，依赖主观估计“约 24 处”。
